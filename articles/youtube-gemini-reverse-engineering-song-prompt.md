# Claude CodeはYouTubeを見られない — Gemini APIで楽曲をリバースエンジニアリングした話

## はじめに

「このYouTube動画と同じ雰囲気の曲をAIで作りたい」。そう思った時、最初にぶつかる壁があります。

**コーディング向けAIエージェント（Claude Code等）は、YouTube動画を視聴・聴取できない。**

テキストは読める、画像は一部見える、しかし動画の音響と映像を時間軸で追って「この曲のサビはどう構成されているか」「ジャケットの画風は何系か」を分析することはできません。

本記事は、この制約を **Gemini API に分析を完全委任するアーキテクチャ** で突破し、YouTube動画から「音楽 / 画像 / 動画」の3モダリティの生成AI用プロンプト仕様書を自動生成した記録です。

- **対象読者**: 生成AI活用者 / プロンプトエンジニア / Claude Codeユーザー / 音楽AIに興味がある人
- **成果物**: YouTube URL → 4セクション構造のプロンプト仕様書（Markdown）を自動出力するスキル

## TL;DR

```
YouTube URL
   │
   ▼
[Claude Code] ── マスタープロンプト組立 ──▶ [Gemini API] ── 真正解析（視聴・視覚）
   ▲                                              │
   │                                              ▼
   └──────── 結果のJSON受領 ◀── {status, summary, full_data}
   │
   ▼
構造検証 → Markdown整形 → 仕様書保存
```

**役割分担の核心**:
- **Claude Code**: プロンプトの組立・結果の構造検証・Markdown整形・ファイル保存
- **Gemini API**: 動画の視聴・視覚による真正解析（CCにはできない領域）

## なぜ「委任」なのか — 得手不得手の境目

最初は「全部Claude Codeでやる」を試しました。ダメでした。理由は明確です。

| 能力 | Claude Code | Gemini API |
|---|---|---|
| YouTube動画の視聴・聴取 | ❌ 不可 | ✅ 可能（マルチモーダル） |
| 長文プロンプトの組立・構造化 | ✅ 得意 | △ |
| 結果のMarkdown整形・検証 | ✅ 得意 | △ |
| ファイル保存・ワークフロー制御 | ✅ 得意 | ❌（ステートレス） |

両者が**自分の得意なことだけを担う**形にすればよい、というのが出発点です。Gemini APIはYouTube動画を直接入力として受け付け、音と映像を時間軸で分析できます。一方、分析結果の「仕様書として体裁を整える・検証する・保存する」はClaude Codeの得意領域です。

この「能力の境目」に沿って処理を分けたのが本アーキテクチャです。

## 実装の中身

### gemini.py — 真正解析の実行部

Gemini APIを叩く薄いラッパーです。核となるのは **「統一JSONで結果を返す」** という約束です。

```python
"""Gemini API経由でYouTube動画を真正解析し、統一JSONで結果を返す.

Usage:
    gemini.py --youtube <URL> [--prompt-file <path>] [--model <name>]
"""
```

実行すると、常にこの形のJSONを標準出力に返します。

```json
{
  "status": "ok",
  "summary": "（解析結果の要約・本文）",
  "full_data": "（詳細データ・キャッシュ参照用）",
  "error": null
}
```

**なぜこの形か**。Claude Code側は `summary` だけを読み込めばよく、巨大な `full_data` は都度コンテキストを圧迫しません。必要な時だけキャッシュファイルから追加で読む、という設計です。

- `status=error` のときはエラー内容をユーザーに提示して手動フローへフォールバック
- `status=ok` のときは `summary` を解析結果として次フェーズへ

### マスタープロンプトの「二重管理」

Geminiに投げる分析指示書（マスタープロンプト）は、以下の4セクションを必ず出力するよう要求します。

1. **楽曲構造・リファレンスデータ**（タイムライン表）
2. **音楽生成AI用プロンプト**（Style/Tags + Lyrics制御タグ）
3. **画像生成AI用プロンプト**（3シチュエーション + Negative Prompt）
4. **動画生成AI用モーション制御**（静寂/動的の2セクション）

このマスタープロンプトは「SSOT（一次ソース）」と「スキル内の実体コピー（二次）」の2箇所で管理しています。symlinkでもよさそうに見えますが、Claude Code環境で `references/` が確実に自動読込される保証がないため、実体コピーを採用しました。変更頻度が低いファイルなので二重管理のコストを許容しています。

### Last.fm で精度を上げる前置き

Geminiにいきなり動画を投げる前に、**Last.fm API で楽曲メタデータを事前取得**するステップを挟みます。

```bash
.venv/bin/python scripts/api/lastfm.py --youtube "<URL>"
```

これで「曲名 / アーティスト / ジャンル・タグ / 類似アーティスト」が分かります。これをマスタープロンプトの末尾に「参考メタデータ」として追記した一時プロンプトファイルを生成し、Geminiに渡します。

メタデータがあると、Geminiの分析精度と再現性が上がります。「この曲は〇〇ジャンルの◎◎アーティストに類似」という補助線が引けるからです。

**失敗時の設計も重要**: Last.fm未登録の楽曲（オリジナル楽曲等）や、動画タイトルのノイズ（「【高画質】」等）で特定に失敗した場合は、メタデータなしでそのままGeminiへ進むようスキップします。壊れない設計です。

## 出力物 — 4セクション構造のプロンプト仕様書

実際に出力される仕様書のイメージ（一般化した例）です。

### セクション1: 楽曲構造・リファレンスデータ

```markdown
| タイム | セクション | 特徴 |
|---|---|---|
| 0:00-0:15 | Intro | シンセパッド+フィルターNs |
| 0:16-0:45 | Verse A | 低音域ボーカル・ミニマル打ち込み |
| 0:46-1:15 | Chorus | 4つ打ち・高音域広がり |
```

### セクション2: 音楽生成AI用プロンプト

```markdown
## Style / Tags
electropop, dreamy, female vocal, 118 BPM, synth pad, four-on-the-floor

## Lyrics 制御タグ
[Verse] ... [Chorus] ... (構造タグでセクション明示)
```

### セクション3: 画像生成AI用プロンプト

```markdown
## シチュエーション1: ジャケット
Prompt: neon cityscape, cyberpunk, purple-pink gradient, ...
Negative Prompt: text, watermark, lowres, ...

## シチュエーション2 / 3: ...
```

### セクション4: 動画生成AI用モーション制御

```markdown
## 静寂セクション: [Static shot] カメラ固定・緩やかな光の揺らぎ
## 動的セクション: [Push in] + [Tracking shot] 手ブレ風演出
```

音楽・画像・動画の**3モダリティを1パスで**仕様化できるのが、この手法の強みです。「曲だけ作って後で映像を考える」と分断されず、一貫した世界観のプロンプトセットが手に入ります。

## ハイブリッド5フェーズ — 人間判断ポイントの設計

全自動ではなく、要所で人間が止まる設計にしています。楽曲・映像は「聴かない/見ないと分からない」ので、AIの独断で進めないためです。

```
Phase 0: 対象確認（URL・粒度）               🔴 人間判断
Phase 0.5: Last.fm メタデータ取得・確認       🔷 確認
Phase 1: Gemini API 自動解析                 （自動）
Phase 2: Gemini結果受領・構造検証             🔴 人間判断
Phase 3: 単独ファイル保存                    （自動）
Phase 4: 他スキルへの橋渡し（任意）           🔴 人間判断
```

- 🔴 は必ず停止してユーザー確認を待つ
- Gemini結果が想定した4セクション構造を満たさなければ、追加指示を与えて再実行

**構造検証のチェックリスト**（Phase 2）:
- [ ] セクション1にタイムライン表があるか
- [ ] セクション2に Style/Tags + Lyrics制御タグがあるか
- [ ] セクション3に3シチュエーション + Negative Promptがあるか
- [ ] セクション4に静寂/動的の2セクションがあるか

この「機械が作ったものを機械が検証する」二段構えで、出力の安定性を担保しています。

## 忘却対策 — スキル化と双方向リンク

ここまでの仕組みは、**「存在を忘れたら機能しない」**というリスクを抱えます。マスタープロンプトをどこかに置いておくだけでは、「あのプロンプト、どこだっけ」で終わります。

そこで2つの仕掛けを入れました。

### 1. トリガーワードで発動する独立スキル化

「逆コンパイルして」「この曲から仕様書作って」「YouTubeから分析して」などの発言で自動発動するスキルにしました。トリガーワードを複数バリエーション用意し、`description` 欄にも類似言い回しを含めて発動率を上げています。

### 2. 双方向リンク（入口と出口）

既存の楽曲制作スキル（例: サイバー和モダン楽曲制作）の入口に、こんな注記を入れています。

> 「外部楽曲をベースにしたい場合 → reverse-engineer-song スキルでYouTube動画から仕様書を逆コンパイルできます」

逆に、この逆コンパイルスキルの出口（Phase 4）では、結果を楽曲制作スキルへ橋渡しするか確認します。**入口から出口へ、出口から入口へ**流れる経路を両方向に張ることで、どのスキルから入っても迷子にならない設計にしました。

## まとめ

**Claude Codeではできないことを、Gemini APIに委任する。** ただそれだけの設計ですが、以下の恩恵があります。

1. **能力の境目で処理を分ける** — 不得手な領域を他LLMに委ね、自分は得手な構造化・保存に集中
2. **統一JSONで結果を受け取る** — `summary` と `full_data` を分離し、コンテキストを圧迫しない
3. **メタデータ前置きで精度UP** — Last.fm で補助線を引いてから解析
4. **3モダリティを1パスで仕様化** — 音楽・画像・動画の一貫したプロンプトセット
5. **人間判断ポイントで止まる** — AIの独断を防ぐ二段検証
6. **忘却対策はスキル化+双方向リンク** — 存在を忘れても経路が導く

「AIエージェントは動画を見られない」は制約ですが、**複数のAIの得意領域を繋ぐ**ことで制約を設計の起点に変えられます。単一の万能AIを待つより、得手不得手を組み合わせる方が、今はずっと実用的です。
