## はじめに

システム開発において、「環境変数の設定忘れ」は非常にありがちなミスです。
APIキーを環境変数から取得するはずが、未設定のまま処理が進んでしまい、エラーが発生するどころか「デフォルト値」や「空文字」が使われてそのまま動き続けてしまう――こんな経験はありませんか？

これは一見「エラーにならずに動いて便利」と思えるかもしれませんが、セキュリティの観点では非常に危険です。重要な暗号化キーが未設定であるにもかかわらずシステムが動き続けると、最悪の場合、データが平文で保存されたり、内部のエラー情報が外部に漏洩したりするリスクがあります。

今回は、私が開発に携わっているOSSプロジェクト「NexusCore」で実際に導入した**fail-closed（フェイルクローズド）設計**を例に、環境変数が未設定のときに「安全に処理を停止させる」ための具体的な実装方法を解説します。

## fail-closed（フェイルクローズド）とは？

まず、セキュリティ設計における2つの考え方を初心者向けに紹介します。

- **fail-open（フェイルオープン）**
  エラーや異常が発生した際、とりあえずシステムを動かし続ける（通す）設計です。利便性は高いですが、セキュリティホールに直結しやすいです。
- **fail-closed（フェイルクローズド）**
  エラーや異常が発生した際、安全を担保するためにシステムを停止させる（閉じる）設計です。

たとえば、認証システムがダウンしたとき、「とりあえず全員を通す（fail-open）」のは恐ろしいですよね。同様に、暗号化のためのキーが設定されていないとき「とりあえず平文で保存する（fail-open）」のも絶対に避けるべきです。

「設定漏れがあるなら、処理を停止させる」というのが、fail-closed設計の基本的な考え方です。

## Pythonでの実装例：環境変数未設定時に処理を止める

NexusCoreでは、ユーザーが入力したAPIキーなどを暗号化してデータベースに保存する処理があります。このとき、暗号化のためのキー（`NEXUS_ENCRYPTION_KEY`）が環境変数に設定されていなかった場合、**保存処理を即座に拒否してエラーを返す**ように実装しています。

以下に、その具体的なPython実装例をシンプルに示します。

```python
import os

class EncryptionKeyMissingError(Exception):
    """暗号化キー未設定時のカスタム例外"""
    pass

def get_encryption_key() -> str:
    """
    環境変数から暗号化キーを取得する。
    未設定または空文字の場合は処理を停止する。
    """
    key = os.getenv("NEXUS_ENCRYPTION_KEY")
    
    # fail-closed: キーが存在しない、または空文字の場合は例外を発生させる
    if not key:
        # セキュリティ上の理由により、エラーメッセージはシンプルに留める
        raise EncryptionKeyMissingError("Encryption key is missing. Operation aborted.")
    
    return key

def save_api_key(user_api_key: str):
    """
    ユーザーのAPIキーを暗号化して保存する
    """
    try:
        # 暗号化キーの取得（未設定ならここで処理が止まる）
        enc_key = get_encryption_key()
        
        # キーを使って暗号化処理を行う
        encrypted_data = encrypt(user_api_key, enc_key)
        
        # データベースに保存
        db.save(encrypted_data)
        print("APIキーを安全に保存しました。")
        
    except EncryptionKeyMissingError as e:
        # ログには例外の詳細（スタックトレースなど）を出力しない
        # 内部情報の漏洩を防ぐため
        print(f"エラー: {e}")
        raise  # 呼び出し元に例外を伝播させ、処理を完全に停止させる
```

### 実装のポイント

1. **`if not key:` による厳格なチェック**
   `os.getenv()`は、環境変数が存在しない場合は`None`を返します。また、開発者が誤って空文字（`""`）を設定していた場合も想定し、`not key`という条件でどちらも弾くようにしています。
2. **カスタム例外の利用**
   汎用的な`Exception`ではなく、専用の例外（`EncryptionKeyMissingError`）を定義することで、「なぜ処理が停止したのか」を呼び出し元で明確にハンドリングできるようにしています。
3. **例外の詳細をログに残さない**
   NexusCoreの最近のコミットでも「APIのログから例外詳細を除去（情報漏洩防止）」という修正が入りました。エラーが起きたとき、スタックトレース（エラーの詳細な履歴）をそのままログに出力してしまうと、攻撃者にシステムの内部構造をヒントを与えてしまいます。そのため、エラーメッセージはシンプルに留めるのがベストプラクティスです。

## 初心者が陥りやすい罠：デフォルト値によるFail-open

初心者の方がやりがちなのが、以下のような実装です。

```python
# 危険な例
KEY = os.getenv("SECRET_KEY", "default_unsafe_key")
```

これは環境変数が未設定のとき、自動的に `"default_unsafe_key"` という値を代入するコードです。ローカルでの開発中はエラーにならないため便利ですが、本番環境にこのままデプロイされると、脆弱なキーを使って暗号化が行われてしまいます。

fail-closed設計においては、**「デフォルト値に逃げない」**ことが重要です。
セキュリティに関わる重要な設定値（データベースのパスワード、暗号化キー、APIトークンなど）については、上記の実装例のように「未設定ならエラーで止める」という強い意志を持たせてください。

## おわりに

システムを開発していると、「エラーが出ない＝良い状態」と思い込みがちですが、セキュリティにおいては「問題があるのにエラーが出ないこと」が最も危険です。

fail-closed設計を取り入れることで、設定ミスによる情報漏洩やデータ破損のリスクを根源から防ぐことができます。「この設定がなかったら、システムは動くべきではない」というラインを明確にし、コード上でしっかりとガードを書くようにしましょう。

今回紹介したテクニックは数行のコードで実装できるため、明日からの開発でもすぐに取り入れられるはずです。ぜひ皆さんのプロジェクトでも、セキュアな設計を意識してみてください！
