## 1. はじめに：AIエージェントの穴、塞いでますか？

ChatGPTやClaudeなどのAPIを組み合わせて、自分専用のAIエージェントを構築する人が増えています。
ローカルPCで動かす分には問題ありませんが、「外からもスマホや別のPCから呼び出したい」となると、VPS（仮想プライベートサーバー）などで公開する必要があります。

しかし、ここに**致命的な落とし穴**があります。

VPSでAIエージェントをそのまま公開してしまうと、悪意のある攻撃者に乗っ取られ、**あなたのAPIキーを使って勝手に高額な請求を発生させる**ことができてしまいます。

この記事では、VPSでAIエージェントを安全にホストするための**「4層防御インフラ」**の設計図を、初心者の方でも真似できるように具体的なコード付きで解説します。

「インフラ周りは苦手…」と敬遠していた方でも「自分でもできそう！」と思えるように、設定をコピペで済ませられる構成にしました。

## 2. 脅威モデル：何から守るべきか？

対策を考える前に、誰から、何から守るべきか（脅威モデル）を整理しましょう。

- **ポートスキャナー**: インターネット上を常に徘徊し、無防備なポート（特にエージェントのデフォルトポート）を探しています。
- **中間者攻撃者**: 通信を盗聴し、あなたのAIとの会話内容やAPIキーを奪おうとします。
- **自動化ボット**: 推測しやすいパスワードや、認証がないURLに対して攻撃を仕掛けてきます。

これらを完全に防ぐためには、**「外側から内側へ」**と複数の壁を重ねる「多層防御」が鉄則です。

## 3. 4層防御アーキテクチャの全体像

今回構築するセキュリティの構造は以下の4つのレイヤー（層）で構成されます。

```text
インターネット
  ↓
[Layer 1] UFW ファイアウォール (不要な通信を足切り)
  ↓
[Layer 2] Caddy TLS/HTTPS (通信の暗号化)
  ↓
[Layer 3] HTTP BasicAuth (URLへのアクセス制限)
  ↓
[Layer 4] Gateway Token + デバイスペアリング (エージェント独自の認証)
  ↓
AIエージェント起動！
```

それでは、1つずつ具体的な設定コードを見ていきましょう。

## 4. レイヤー1: UFW ファイアウォール（第一の壁）

まずはOSレベルで外部からの不要な通信を遮断します。AIエージェントが使用するポート（例: 18789）は、絶対に外部に直接公開してはいけません。

VPSサーバー上で以下のコマンドを実行し、UFWを設定します。

```bash
# デフォルトで外部からの通信をすべて拒否
sudo ufw default deny incoming
# 外部への通信はすべて許可
sudo ufw default allow outgoing

# Web通信（HTTPS）と証明書取得用（HTTP）のみ許可
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# SSH接続を許可（ポート番号22は自身の環境に合わせてください）
# limitオプションにより、ブルートフォース攻撃を自動遮断します
sudo ufw limit 22/tcp

# エージェントのポートは外部からアクセスできないよう明示的に拒否
sudo ufw deny 18789

# 設定を有効化
sudo ufw enable
```

### 💡 ポイント：ローカルバインディング
UFWで拒否するだけでなく、AIエージェント（またはDocker）が起動する際のポートを、外部から見えない `127.0.0.1`（ローカルホスト）のみにバインドします。

Dockerを使用する場合は、ポートマッピングを以下のように設定します。
```yaml
ports:
  - "127.0.0.1:18789:18789" # 左辺に127.0.0.1を付けるのが超重要！
```

## 5. レイヤー2: Caddy TLS/HTTPS（第二の壁）

AIエージェントとの通信を暗号化し、盗聴を防ぎます。通常、SSL/TLS証明書の取得と更新は面倒ですが、Webサーバーである**Caddy（キャディ）**を使うと、すべて自動で完了します。

Caddyの設定ファイル（`Caddyfile`）は驚くほどシンプルです。

```caddyfile
fopenclaw.com {
    encode gzip

    # Caddyが自動でLet's Encrypt証明書を取得・更新してくれます
    reverse_proxy localhost:18789
}
```

設定ファイルにドメイン名を書くだけで、Caddyが勝手にHTTPS化（TLS終端）をしてくれます。運用コストは**ゼロ**です。

## 6. レイヤー3: HTTP BasicAuth（第三の壁）

たとえHTTPSで暗号化されていても、URLを知っていれば誰でもアクセスできる状態は危険です。そこで、URLにアクセスするための「IDとパスワード」を要求するBasicAuthを設けます。

まずは、Caddyのコンテナを使って安全にパスワードのハッシュ値を生成します。

```bash
# 好きなパスワードを入力してハッシュ値を生成
echo "あなたの強力なパスワード" | docker run -i --rm caddy:2 hash-password
# 出力例: $2a$14$xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

生成されたハッシュ値を使って、先ほどの `Caddyfile` をアップデートします。

```caddyfile
fopenclaw.com {
    encode gzip

    # BasicAuthの設定
    basicauth {
        # ユーザー名と先ほど生成したハッシュ値を記述
        deployer $2a$14$xxxxxxxxxxxxxxxxxxxxxxxxxxxxx
    }

    reverse_proxy localhost:18789
}
```

これで、URLにアクセスするとブラウザにIDとパスワードの入力画面が表示されるようになり、自動化ボットや第三者を弾くことができます。

## 7. レイヤー4: Gateway Token + デバイスペアリング（第四の壁）

最後はアプリケーション層の防御です。万が一、BasicAuthを突破されても、AIエージェント自身が持つ認証機能で攻撃を弾きます。

エージェントの設定ファイル（例: `config.json`）で、通信に必要なトークン（鍵）と、アクセスを許可するデバイスを制限します。

```json
{
  "gateway": {
    "auth": {
      "mode": "token",
      "token": "${GATEWAY_TOKEN}"
    }
  },
  "plugins": {
    "device-pair": {
      "config": {
        "publicUrl": "https://fopenclaw.com"
      }
    }
  }
}
```

このように設定しておくことで、あなたが事前にペアリング（紐付け）を許可したデバイス以外からは、AIエージェントのAPIを叩くことができなくなります。

## 8. おまけ：ローカルPC構成との比較

「そもそもVPSで公開せず、自宅のローカルPCで運用したほうが安全では？」と思う方もいるでしょう。結論から言うと、ローカルPC運用も十分強力ですが、セキュリティのベクトルが異なります。

| 比較項目 | VPSでのホスティング（本記事の構成） | 自宅ローカルPCでのホスティング |
| --- | --- | --- |
| **強力な点** | 外部からの攻撃に強い（多層防御） | 家庭のルーターによる NAT/ネットワーク分離が壁になる |
| **インターネット露出** | グローバルIPに直接来る攻撃を4層で弾く | 基本的に外部からは見えない（TailscaleなどのVPNを使用） |
| **運用のポイント** | Caddyによる証明書の自動更新が必須 | 自宅のIPアドレス変更や、PCの常時起動の管理が必要 |

自宅PCをTailscaleなどのVPNに繋いで運用する方法も非常に安全ですが、「外からいつでも安定して呼び出したい」という用途であれば、今回解説した**VPSでの4層防御**が最も堅牢で現実的な選択肢です。

## 9. まとめ

AIエージェントを安全にVPSで運用するための4層防御について解説しました。

1. **UFW**で通信を絞る
2. **Caddy**で暗号化を自動化
3. **BasicAuth**でURLをガード
4. **Token + ペアリング**でエージェント自体をロック

セキュリティと聞くと難しく感じますが、設定自体は数十行のコピペで終わります。AIを外の世界に公開する際は、ぜひこの4層防御をベースに安全なインフラを構築してみてください！
