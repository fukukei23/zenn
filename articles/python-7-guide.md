## はじめに

テストを書く際、皆さんはどのようにテストケースを思いついているでしょうか？

おそらく「正常な入力」「境界値（0や空文字など）」「異常な入力」といったパターンを経験則で考え、`assert func(2) == 4` のように具体的な例をいくつか挙げてテストを書いているのではないでしょうか。これを**例示テスト（Example-Based Testing）**と呼びます。

例示テストは直感的でわかりやすい反面、「自分が思いつかなかった不具合」を見逃してしまうリスクを常に抱えています。

最近、私は音楽データを処理するツールの開発において、Pythonの `Hypothesis` というライブラリを用いた**プロパティベーステスト（Property-Based Testing）**を本格的に導入しました。数千回のランダムなテストを自動生成することで、手動では絶対に思いつかなかったようなエッジケースのバグをいくつも発見できました。

本記事では、プロパティベーステストの基本的な考え方から、実際に私が実務で実装した7つのテストケース（対称性・単調性・不変量など）の中から、特に初心者向けの重要な概念を具体的なPythonコードとともにわかりやすく解説します。

## 例示テストからプロパティベーステストへ

プロパティベーステストとは、具体的な「入力と出力の組み合わせ」を書くのではなく、「**どんな入力を与えても、この関数は常にこの性質（プロパティ）を満たすべきだ**」というルールを記述するテスト手法です。

Pythonのデファクトスタンダードなライブラリである `Hypothesis` を使うと、このルールに基づいて数百〜数千のランダムなデータを自動生成し、テストを実行してくれます。

従来のテスト：
```python
def test_add():
    assert add(1, 2) == 3
    assert add(-1, 1) == 0
```

プロパティベーステスト（Hypothesis）：
```python
from hypothesis import given, strategies as st

@given(st.integers(), st.integers())
def test_add(a, b):
    # 足し算の順序を入れ替えても結果が同じであるという「性質」をテスト
    assert add(a, b) == add(b, a)
```

このように、「具体的な正解」を書く代わりに、「関数が満たすべき性質」を宣言します。これにより、人間の想像力を超えた広範な入力パターンを網羅できます。

## 実務で使える3つのプロパティパターンと実装例

私が実際に開発した音声解析ツール（yt-dlpで音声を取得し、basic_pitchで解析するツール）のテスト戦略において、以下のようなプロパティを定義しました。ここでは初心者にも応用しやすい3つのパターンを紹介します。

### 1. 不変量・範囲
「関数にどんな入力を与えても、出力は常に特定の範囲内に収まる」という性質です。データの正規化処理などで非常に強力に機能します。

例えば、音声データのノイズレベルを計算する関数 `calculate_noise_level` は、常に `0.0` から `1.0` の範囲でパーセンテージを返さなければなりません。

```python
import hypothesis.strategies as st
from hypothesis import given

@given(st.floats(min_value=-10000.0, max_value=10000.0, allow_nan=False))
def test_noise_level_is_always_between_0_and_1(raw_audio_value):
    # どんな極端な数値を入れても、結果は0.0〜1.0の間でなければならない
    result = calculate_noise_level(raw_audio_value)
    assert 0.0 <= result <= 1.0
```

### 2. 単調性
「入力Aが入力Bより大きい（または小さい）場合、出力も同じように大小関係を保つ」という性質です。スコア計算や優先度のソートなどでよく使われます。

音量が大きくなるほど、検出スコアも高くなる（または同じになる）べき、といったテストです。

```python
@given(
    low_volume=st.floats(min_value=0.0, max_value=0.5),
    high_volume=st.floats(min_value=0.5, max_value=1.0)
)
def test_score_is_monotonic(low_volume, high_volume):
    # high_volumeはlow_volumeより常に大きい
    assert low_volume <= high_volume
    
    score_low = calculate_detection_score(low_volume)
    score_high = calculate_detection_score(high_volume)
    
    # スコアも大小関係を維持しているべき（単調増加）
    assert score_low <= score_high
```

### 3. 対称性（双方向の変換）
「データを変換して、再度逆変換で元に戻せる」という性質です。エンコード/デコード処理や、暗号化/復号処理などで必ず確認すべきプロパティです。

```python
@given(st.text()) # あらゆるランダムな文字列
def test_encode_decode_symmetry(original_text):
    # エンコードして、デコードしたら元のテキストに戻るはず
    encoded = encode_data(original_text)
    decoded = decode_data(encoded)
    
    assert decoded == original_text
```

実務では、この他にも「結合則（計算の順序を変えても結果が同じ）」や「冪等性（何度実行しても同じ結果になる）」など、合計7つのプロパティベーステストを定義し、ツールの堅牢性を劇的に向上させました。

## 失敗時のデバッグが超絶便利

プロパティベーステストの魅力は、バグを見つける能力の高さだけでなく、**「バグを見つけたときの報告のわかりやすさ」**にもあります。

もしランダムな10万通りのテストデータのうち、特定の入力でテストが失敗した場合、「どの入力で失敗したか」を探し出すのは困難に思えるかもしれません。しかし、Hypothesisは失敗を検知した瞬間、**そのテストを失敗させる「可能な限り最小の入力データ」を自動で逆算（シュリンク）してくれます。**

例えば、極端に巨大な浮動小数点でエラーが起きた場合でも、Hypothesisは「入力が `0.0001` のときにエラーになります」というように、デバッグしやすい形にまで問題を縮小してレポートしてくれます。

また、失敗したデータはキャッシュされ、次回からは必ずそのエッジケースからテストされるようになるため、デグレ（回帰バグ）の防止にも絶大な効果を発揮します。

## おわりに

従来の例示テストを完全に置き換える必要はありません。重要なビジネスロジックや、複雑なデータ処理を行う純粋関数に対して、プロパティベーステストを「上乗せ」していくアプローチが現実的です。

最初は「どんな性質をテストすればいいのか」を考えるのに少し慣れが必要かもしれませんが、一度コツを掴めば、アプリケーションの堅牢性が飛躍的に向上するのを実感できるはずです。

皆さんも自分のプロジェクトのユニットテストに、さっそく `Hypothesis` を1つ導入して、プロパティベーステストの世界を体験してみてはいかがでしょうか。
