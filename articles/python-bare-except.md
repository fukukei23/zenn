## はじめに

Pythonでコードを書いていると、思わぬエラーでプログラムが落ちるのを防ぐために `try...except` を使う場面が多くあります。しかし、例外を捕捉しすぎるあまり、本来気づくべきバグやシステム障害まで隠してしまってはいないでしょうか。

私が開発に参加しているプロジェクト「NexusCore」でのRound 2一気通貫リファクタリングにおいて、**bare except（ベアエグセプト）の完全排除**に取り組みました。この作業を通じて、38ファイル・約50カ所に及ぶエラー処理を安全な形（typed exceptions）へと置き換えました。

本記事では、なぜbare exceptがセキュリティや安定性の脅威になるのかという理論的な背景から、実際のプロジェクトでどのように修正を行ったのかという具体的な手順までを解説します。初心者の方でも「より安全なPythonコードの書き方」として参考にしていただければ幸いです。

## 1. なぜbare exceptが脅威となるのか？

まず、bare exceptとは以下のように例外の種類を指定せずにすべての例外をキャッチする書き方を指します。

```python
# やってはいけない bare except の例
try:
    config = load_config()
except:
    config = default_config
```

一見、「エラーが出てもデフォルト値で動き続けるから頑丈だ」と思えるかもしれません。しかし、これには重大なリスクが潜んでいます。

### プログラムが停止しなくなる問題
Pythonでは `KeyboardInterrupt`（Ctrl+Cによる中断）や `SystemExit`（`sys.exit()`による終了）、`MemoryError`（メモリ不足）など、**プログラムを緊急停止させるべき重大な例外**も存在します。bare exceptはこれらまで捕捉してしまい、深刻なエラーが起きているのにシステムがゾンビのように動き続ける原因になります。

### STRIDE分析による脅威の特定
今回のリファクタリングでは、設計判断の理由を明確にするために**STRIDE分析**（なりすまし、改ざん、否認、情報漏えい、サービス妨害、権限昇格）を用いた脅威モデリングを実施しました。

エラーを適切にハンドリングせず握りつぶすことは、STRIDEのうち**「情報漏えい」**や**「サービス妨害」**、さらにはエラー発生源が追えないことによる**「否認」**のリスクに直結します。ログに何も残されずに処理が継続されることで、攻撃者の探索活動や内部システムの異常動作に気づけなくなるのです。

## 2. typed exceptionへの置き換え具体的手順

STRIDE分析でリスクを特定した私たちは、Phase 3としてプロジェクト内（webapp, llm, core, agentsなど）のすべてのbare exceptを **typed exceptions（型指定された例外）** に置き換える作業を実施しました。

具体的には、発生し得る例外を特定し、それぞれに適切な処理を割り当てていきます。以下に代表的な修正パターンを紹介します。

### パターン1: 例外を明示的に指定する
最も基本的な修正です。想定されるエラー（例：ファイルが存在しない、フォーマットが不正など）を明示的に指定します。

```python
# 修正前: すべてのエラーが握りつぶされる
try:
    user_input = int(input_data)
except:
    user_input = 0

# 修正後: 想定される例外のみをキャッチする
try:
    user_input = int(input_data)
except ValueError as e:
    logger.warning(f"Invalid input format: {e}")
    user_input = 0
```

### パターン2: 致命的なエラーはログを出して再送出する
システムの継続が困難なエラーまで捕捉して処理を続けると、データ不整合などを引き起こします。この場合は例外を再度発生させる `raise` を使用します。

```python
import logging

logger = logging.getLogger(__name__)

# 修正後: 致命的なエラーはキャッチせずそのままクラッシュさせるか、
# ログだけ残して上位に処理を任せる
try:
    db_connection.connect()
except ConnectionError as e:
    logger.error(f"Database connection failed: {e}")
    # 代替手段がない場合はそのまま例外を投げる
    raise
except Exception as e:
    # 予期せぬエラーが起きた場合の緊急脱出ログ
    logger.critical(f"Unexpected error: {e}", exc_info=True)
    raise
```

`except Exception as e:` は `BaseException` を継承するシステム終了系の例外（`KeyboardInterrupt`など）は捕捉しないため、bare exceptよりも安全です。

## 3. 大規模リポジトリでの進め方と工夫

NexusCoreのような複数のモジュール（API、CLI、設定、オーケストレーターなど）を持つ大規模リポジトリでは、一気にすべてを修正するとデグレ（回帰バグ）の温床になります。今回私たちが実施した安全な進め方の手順を紹介します。

### Step 1: 対象の網羅的洗い出し
`grep` コマンドや静的解析ツール（Flake8やRuffなど）を使用して、リポジトリ内のすべてのbare except（`except:` や `except Exception:` の不適切な使用）を洗い出しました。

### Step 2: 影響範囲ごとの分割適用
約50カ所の修正を一つのPRにまとめるのではなく、モジュールごとに細かく分割して適用しました。
- Phase 3-1: `webapp` + `llm` モジュール
- Phase 3-2: `core` / `orchestrator` / `utils` 群
- Phase 3-3: `agents` / `api` / `workflows` 群

このように小さく区切ることで、各修正が他の機能に悪影響を与えていないかテストしやすくなります。

### Step 3: テストによる動作保証
コードを修正しただけでは「本当にそのエラーが捕捉されるのか」を保証できません。実際にNexusCoreでは、テストコード内でも例外が正しく送出されるかを確認するユニットテストを追加しました。結果として、全1391テストが通過（0 failed）し、安全にリファクタリングが完了しました。

## おわりに

エラー処理は「とりあえずエラーが出ないようにする」ためのものではなく、「異常をいち早く検知し、安全にシステムを停止または復旧させる」ためのものです。

bare exceptの排除は、一見すると地味な作業ですが、システム全体の堅牢性を高め、保守性を劇的に向上させます。STRIDE分析のようなセキュリティの観点から見ても、脅威を特定・修正するための重要な基盤となります。
