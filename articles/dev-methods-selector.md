## はじめに

「どの開発手法を使えばいいんだろう？」

新人エンジニアの頃、こんな疑問を持ったことがある方は多いのではないでしょうか。TDD、BDD、DDD、クリーンアーキテクチャ、仕様駆動開発——言葉は聞いたことがあるけど、いつ使うべきか、どう選んだらいいのかわからない。

そんな方のために、**プロジェクトの種類から最適な手法を選ぶ早見表**を作りました。

## プロジェクトの種類を見極める

まず、自分のプロジェクトがどのタイプに当てはまるかを確認しましょう。

| プロジェクトのタイプ | 特徴は？ |
|---|---|
| **スクリプト・CLI** | スクレイピング、CLIツール、データ変換バッチなど |
| **Webアプリ（要件明確）** | REST API、Webアプリバックエンドなど仕様がある程度決まっている |
| **Webアプリ（複雑）** | 予約システム、在庫管理、ワークフローなどビジネスルールが多い |
| **ライブラリ** | npmパッケージ、Pythonパッケージなど外部に公開するもの |
| **プロトタイプ** | アイデア検証、ハッカソンなど動くものを作りたい段階 |

## 5つのレシピ

### レシピA: 軽量開発 — スクリプト・CLI向け

**対象**: スクレイピング、CLIツール、データ変換バッチ

| 手法 | 必須/推奨 |
|---|---|
| 設計原則（DRY・KISS・YAGNI） | **必須** |
| TDD | 推奨 |
| CI/CD | 推奨 |

複雑な手法は不要。KISS（シンプルに書く）を意識すれば十分です。

### レシピB: 規格駆動開発 — Web API向け

**対象**: REST API、Webアプリバックエンド、既存仕様あり

| 手法 | 必須/推奨 |
|---|---|
| 仕様駆動開発（OpenAPI等） | **必須** |
| TDD | **必須** |
| 設計原則（SOLID等） | **必須** |
| CI/CD | 推奨 |

まずAPI仕様を書き、それに基づいてテストを書き、実装するという流れです。

### レシピC: ドメイン中心開発 — 複雑アプリ向け

**対象**: 予約システム、在庫管理、ワークフローなど

| 手法 | 必須/推奨 |
|---|---|
| DDD | **必須** |
| クリーンアーキテクチャ | **必須** |
| TDD | **必須** |
| BDD | 推奨 |
| 設計原則 | **必須** |
| CI/CD | **必須** |

ビジネスロジックが複雑なシステムは、ドメインモデルを先に設計することが重要です。

### レシピD: 品質重視開発 — ライブラリ向け

**対象**: 公開ライブラリ、npmパッケージ、Pythonパッケージ

| 手法 | 必須/推奨 |
|---|---|
| TDD | **必須** |
| 設計原則（ISP等） | **必須** |
| CI/CD | **必須** |

外部に公開するライブラリは、後方互換性を保ちながら機能追加する必要があります。

### レシピE: 高速開発 — プロトタイプ向け

**対象**: アイデア検証、ハッカソン

| 手法 | 必須/推奨 |
|---|---|
| 設計原則（KISSのみ） | 参考 |

テストもCI/CD也不要。**動くものを最速で作った方がフィードバック得快い**です。

## まとめ

開発手法は「全部使わないとけない」するものではありません。**プロジェクトの種類に合わせて、必要なものを選ぶ**ことが大切です。

まずは自分が作ろうとしているものがどのタイプかを見極めて、それに合ったレシピを選んでみてください。
