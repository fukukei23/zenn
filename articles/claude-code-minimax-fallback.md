## はじめに

以前、Zennにて「Claude Code CLIをGLM/Z.AIで代替した話」という記事を公開しました。

https://zenn.dev/fukukei23/articles/claude-code-cost-optimization

この記事では、Claude Code CLIの通信先をZ.AI（GLM）に切り替えることで、コストを大幅に抑えつつ開発を進める手法を紹介しました。実際に運用してみて、Z.AI Pro（GLM-5.1）はコストパフォーマンスが非常に高く、日常的な開発タスクの多くを十分にこなしてくれます。

しかし、運用を続ける中で「Z.AIだけでは不十分な場面がある」ことに気づきました。具体的には、**使用量の上限に達してしまう**ケースや、**一時的なエラーで開発フローが止まってしまう**ケースです。

そこで本記事では、MiniMax Plusをフォールバック（予備系）として組み込み、安いLLMを2段構えで運用する戦略について解説します。

## なぜフォールバックが必要なのか（3つの理由）

Z.AI（GLM）をメインに据えた運用は非常に快適ですが、単一プロバイダーに依存する以下の3つの課題がありました。

### 理由1: 使用量オーバーフロー対策

Z.AIのプランには使用量上限があります。

| プラン | 月額料金 | 使用量（Claude Pro比） |
|---|---|---|
| Lite | $18/月 | 約3倍 |
| Pro | $72/月 | 約15倍 |
| Max | $160/月 | 約60倍 |

開発に没頭していると、特にProプランで運用している場合、上限に達することがあります。問題は、**上限に達した時のデフォルトの挙動**です。何も設定していなければ、Claude Code CLIは本来の通信先であるClaude APIにフォールバックしてしまいます。

Claude APIは最高性能ですが、その分高額です。月額$200の請求が来るリスクは、コスト最適化の観点からは本末転倒です。

ここでMiniMax Plus（月額$20）に逃がすことで、安価なまま開発を継続できます。

### 理由2: エラーレジリエンス

どんなプロバイダーも、一時的な障害は起きます。GLMが一時的に応答できない場面は稀ですが、ゼロではありません。

実際に運用していて、1ヶ月に数回程度はタイムアウトやサーバーエラーに遭遇しました。頻度は高くないものの、**開発フローが止まるとストレス**になります。特に調子よくコーディングを進めている最中にエラーで手が止まるのは、コンテキストの観点でも痛手です。

自動的にMiniMaxに切り替えることで、エラー発生時も開発の継続性を担保できます。

### 理由3: タスクの重み付け

3つ目の理由は、やや異なる観点です。**全てのタスクに高性能モデルは不要**だということです。

開発作業の中には、以下のような「簡単な定型処理」が含まれます。

- テストコードの生成
- 翻訳
- フォーマット変換
- ドキュメントの自動生成
- Lintエラーの修正

これらは、コーディング性能スコアが少し低くても十分にこなせるタスクです。

| モデル | コーディング性能スコア |
|---|---|
| Claude Opus | 47.9 |
| GLM-5.1 | 45.3 |
| MiniMax | 41.9 |

一方で、以下のような「複雑な推論」には高性能なモデルが必要です。

- 設計判断
- バグの原因調査
- アーキテクチャの決定
- リファクタリングの方針検討

コストに見合ったモデル振り分けを行うことで、効率的な開発が可能になります。「安いだけならGLMの高いプランにすればいい」という意見もあるかもしれませんが、タスクに応じてモデルを振り分ける方が、トークン単価の観点からも賢い戦略だと考えています。

## フォールバックの仕組み

Claude Code CLIは、API通信先を環境変数で切り替え可能な仕組みを持っています。これを利用して、以下のようなフローを実現しました。

```
リクエスト送信
    ↓
エラー検知 or 使用量上限?
    ↓ Yes
フォールバック先へ自動切り替え
    ↓
MiniMax Plusで処理継続
```

この仕組みを実現するスクリプト（`claude_fallback.py`）をGitHubで公開しています。本記事では概念の解説に留めますが、具体的な設定手順などはリポジトリのREADMEを参照してください。

https://github.com/fukukei23/claude-cost-optimizer

## 実際の運用構成

現在の運用構成は以下の3層構成になっています。

```
┌─────────────────────────────────────────┐
│  Layer 1: Z.AI Pro（GLM-5.1）           │
│  用途: 複雑なタスク・通常開発             │
│  月額: $72                               │
├─────────────────────────────────────────┤
│  Layer 2: MiniMax Plus                   │
│  用途: 使用量上限時・エラー時・定型処理    │
│  月額: $20                               │
├─────────────────────────────────────────┤
│  Layer 3: Claude API（最終保護）          │
│  用途: どうしても必要な時だけ              │
│  月額: 従量課金（$200リスクあり）          │
└─────────────────────────────────────────┘
```

体感として、Layer 1で全体の80〜90%のタスクを処理し、Layer 2が10〜19%を担い、Layer 3は月に数回程度しか発動しません。この構成により、**コストを抑えつつ開発継続性を高いレベルで担保**できています。

## おわりに

フォールバック戦略により、単一プロバイダーの課題（使用量上限・エラー・コスト）を解決しつつ、開発体験を損なわずに運用できています。

「安いLLMを使う = 品質を下げる」ではなく、「タスクに応じて適切なモデルを選ぶ = コストと品質のバランスを最適化する」という考え方で取り組んでいます。
