# はじめに

「昔からある巨大な関数を修正することになり、コードを開いたら199行もあって絶望した……」

こうした経験は、ソフトウェアエンジニアであれば一度や二度はあるのではないでしょうか。私が現在携わっている `NexusCore` というリポジトリでも、機能追加を重ねるうちに技術的負債が蓄積し、一部の関数が200行近くに肥大化してしまうという事態が発生しました。

今回は、NexusCoreで実施した大規模リファクタリングの実例をもとに、**「199行の巨大関数を57行までスリム化した具体的な思考プロセスと手順」**を解説します。

「リファクタリングが必要だと分かっていても、どこから手をつければいいか分からない」という初心者エンジニアに向けた、実践的なガイドラインになれば幸いです。

---

# 1. リファクタリングの鉄則：動く状態を崩さない

本題のコード分割に入る前に、大前提として押さえておくべきことがあります。それは**「振る舞いを変えずに、内部構造を整理する」**ということです。

NexusCoreのリポジトリでも、長関数の分割（Phase 1）に着手する前に、以下のような地道な改善を複数ファイルにわたって実施しました。

- **未使用importの削除**
- **ハードコードされた値の定数化**
- **`bare except`（例外を無差別にキャッチする記法）の全廃**
- **テストカバレッジの向上（82%→95%など）**

特に `bare except` の修正とテストの追加は重要です。既存の例外処理が雑に書かれていると、関数を分割した際に予期せぬバグを引き起こす可能性があります。まずは**安全にコードを触れる足場（テスト）を固める**ことから始めるのが、失敗しないリファクタリングのコツです。

---

# 2. 巨大関数を分割する具体的な3ステップ

足場が固まったら、いよいよ長関数の分割に取り掛かります。199行の関数を57行にした手順は、大きく以下の3ステップに分かれます。

## ステップ1: コードの「意味的な塊（ブロック）」を見つける

長関数の中には、必ず「ここは〇〇の処理をしているな」という意味的なまとまりがあります。多くの場合、**コメント**や**空行**で区切られていることが多いです。

例えば、NexusCoreのあるAPIルーティング関数（159行）は、次のようなブロックに分かれていました。

1. リクエストのバリデーション
2. DBからのデータ取得
3. 文字列（プロンプト）の組み立て
4. 外部APIへのリクエスト送信
5. レスポンスのフォーマットと返却

## ステップ2: 塊ごとに「関数抽出」を行う

見つけたブロックを、それぞれ独立した関数として切り出します（**関数の抽出**）。
Pythonで具体的なイメージを見てみましょう。

**【Before: 抽出前の長関数（イメージ）】**
```python
def process_user_request(user_id: str, request_data: dict):
    # 1. バリデーション (約20行)
    if not request_data.get("name"):
        raise ValueError("Name is required")
    # ... 多くのバリデーション処理 ...

    # 2. データ取得 (約30行)
    user = db.query(User).filter_by(id=user_id).first()
    # ... データの加工処理 ...

    # 3. プロンプトの組み立て (約40行)
    prompt = f"ユーザー: {user.name} からのリクエストです...\n"
    # ... 複雑な文字列組み立て ...
    
    # 4. API送信と返却 (約20行)
    response = external_api.send(prompt)
    return format_response(response)
```

このようなコードを、メインの関数は**「データの流れを制御するオーケストレーター（指揮者）」**として扱い、各ブロックを別関数に切り出します。

**【After: 抽出後のスッキリした関数】**
```python
def process_user_request(user_id: str, request_data: dict):
    # メインの関数は、全体の流れ（オーケストレーション）だけを記述する
    validate_request(request_data)
    
    user = fetch_user_data(user_id)
    
    prompt = build_api_prompt(user, request_data)
    
    return send_and_format_response(prompt)

def validate_request(data: dict):
    # バリデーションのロジックのみを記述
    ...

def build_api_prompt(user: User, data: dict) -> str:
    # 文字列組み立てのロジックのみを記述
    ...
```

これにより、メイン関数はわずか数行〜数十行になり、「この関数が全体として何をしているか」が一目で分かるようになります。

## ステップ3: 分割単位の責務を確認する

関数を切り出したら、それぞれの関数の**名前**と**処理内容**が一致しているか確認します。
「データ取得」という名前の関数の中に「文字列組み立て」が混ざっていたら、それはまだ分割が不足しています。

NexusCoreのリファクタリングでは、API/ルーティング系の関数や文字列組み立ての関数を対象に、このステップを徹底しました。その結果、複数のファイルにわたって長関数が劇的に短くなり、最も大きな関数は199行から57行までスリム化されました。

---

# 3. 大規模リファクタリングを挫折しないためのコツ

最後に、長期間にわたるリファクタリングをやり遂げるための実務的なコツを2つ紹介します。

### フェーズを分けて少しずつ進める
NexusCoreのコミット履歴を見ると、`Phase 1`、`Phase 2`...というように、タスクを細分化して着実に進めていることが分かります。
- Phase 1: 文字列組み立て関数の分割
- Phase 2: API/ルーティング関数の分割
- Phase 3: bare exceptの全廃

一度に全てをリファクタリングしようとすると、変更箇所が膨大になり、バグの原因特定すら難しくなります。まずは「影響範囲が限定的で、効果が高い箇所（クイックウィン）」から始めるのがおすすめです。

### 機械的な処理で品質を担保する
未使用のimport削除や、定数の切り出しなどは、人の目でやると見落としが発生します。NexusCoreでは、RuffなどのLinterツールを活用しながら、機械的に検知・修正できる部分はツールに任せています。人間の脳のリソースは、コードの分割や設計といったより高度な判断に注ぎましょう。

---

# おわりに

199行の関数を57行にする魔法はありません。あるのは「処理の責務を見極め、少しずつ切り出していく」という地味ですが確実な作業だけです。

長関数に直面して絶望したときは、「まずはテストを書いて足場を固める」「意味の塊を見つけて関数を抽出する」という2つのステップを思い出してみてください。コードが分割されてスッキリしていく快感は、エンジニアとしての大きなモチベーションになるはずです。

この記事が、あなたのプロジェクトのリファクタリングを進める一助になれば幸いです。
