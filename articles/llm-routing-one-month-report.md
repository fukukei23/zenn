## トラブル事例と対策

1ヶ月の運用で遭遇したトラブルと、どう解決したかをまとめます。

### トラブル1: 429レートリミット（約3時間の停止）

**最大のインシデント**は、Z.AI Maxのレートリミット到達による約3時間の完全停止でした。

Z.AI Maxプランの制限は「5時間ウィンドウ1,600プロンプト」「週間8,000プロンプト」です。開発に没頭していると、特にピーク時間帯（15:00〜19:00 JST）はGLM-5.1が通常の3倍の消費量になるため、あっという間に上限に達します。

Claude Code CLIにはAPIレベルのフォールバック機構がありません。429が返ると、**開発フローが完全に停止**します。

**対策**: ローカルレートプロキシ（glm-rate-proxy）を開発・導入しました。

```
Claude Code → localhost:8787（プロキシ）→ Z.AI API / MiniMax API
```

このプロキシは、Z.AIの使用率に応じて自動的にモデルを切り替えます。

| 使用率 | モード | モデル | 消費 |
|---|---|---|---|
| <80% | Normal | GLM-5.1 | 3x（ピーク）/ 2x（オフピーク） |
| 80-95% | Economy | GLM-4.7 | 1x |
| >95% | Emergency | GLM-4.7-Flash | 無料 |

使用率が高くなると、自動的に消費の少ないモデルに切り替わり、429到達を未然に防ぎます。429が発生した場合でも、GLM-4.7-Flash（無料枠）→ MiniMax M2.7 の順でフォールバックするため、開発が止まることはなくなりました。

**結果**: プロキシ導入後、429による停止はゼロです。

### トラブル2: プロキシ起動失敗時の通信断

プロキシはClaude Codeのセッション開始時にフックスクリプトで起動しますが、まれに起動に失敗することがありました。この場合、`ANTHROPIC_BASE_URL`が`localhost:8787`を指したままプロキシがいないため、**全ての通信が失敗**します。

**対策**: フェイルセーフ機構を実装しました。

- プロキシ起動成功 → `settings.json`のURLを`localhost:8787`に変更
- プロキシ起動失敗 → `settings.json`をZ.AI直結URLのまま保持（変更しない）

これにより、プロキシが起動しなくてもZ.AI直結で動作し続けます。

### トラブル3: MCPツール定義のコンテキスト圧迫

LLMルーティングとは直接関係ありませんが、運用上の大きな課題でした。

7つのMCPサーバーが提供するツール定義が、会話コンテキストの約10%（21.4kトークン）を占有していました。これにより、長時間のセッションではコンテキスト不足に陥りやすくなります。

**対策**: 使用頻度の低いMCPサーバー（linear, sentry, supabase, tavily, stripe）を削除し、6サーバーに削減。約35kトークンの節約になりました。

また、設定ファイルが2つ（`~/.claude.json`と`~/.claude/settings.json`）存在し、両方がマージされてロードされることに気づかず、削除が反映されないトラブルも経験しました。Claude Code CLIの`claude mcp remove`コマンドで両ファイルから一括削除できるため、手動編集よりCLIの利用をおすすめします。

## コストに見合っているかの判断

$180/月（約28,000円）で、これだけの開発成果を出せているのは**費用対効果が非常に高い**と判断しています。

同じことをClaude公式プランで行う場合、Max（20x）の$220でも使用量が不足する可能性があります。従量課金で補完すると、月に$500〜$1,000以上かかることも珍しくありません。

個人開発者にとって「いくら使っても月額固定」の安心感は、開発の継続性に直結します。「コストを気にしてAIに任せるのをためらう」という状況をなくすことが、最も大きな効果だったと感じています。

## 今後の改善予定

1ヶ月運用して見えた課題と、今後の改善予定です。

### レートプロキシの安定性監視

現在、プロキシのステータス確認は手動（`curl http://127.0.0.1:8787/proxy/status`）です。将来的には、自動的なヘルスチェックとアラートを組み込む予定です。

### トークン使用量の可視化

Z.AIダッシュボードで全体の使用量は確認できますが、「どのプロジェクトでどれだけ消費したか」の内訳は把握できていません。プロジェクト別の使用量トラッキングを導入したいと考えています。

### MiniMaxの自動委譲の強化

現在はGLMがタイムアウトした際にMiniMaxにフォールバックしていますが、「要約・テスト生成・フォーマット変換」のような定型処理は、最初からMiniMaxに振り分ける方がトークン単価的に効率的です。タスクの性質に応じた事前ルーティングの導入を検討しています。

## まとめ

1ヶ月間、27億トークンを処理したLLMルーティングの実運用から、以下が分かりました。

- **マルチプロバイダー構成は有効**: 単一プロバイダーの429や障害に強い
- **ローカルレートプロキシが鍵**: 使用率に応じた自動モデル切替で429を予防
- **$180/月で十分な開発が可能**: 個人開発の規模なら、公式プランより安く済む
- **フェイルセーフ設計が重要**: プロキシが壊れても直結で動き続ける設計が必須

「安いLLMを使う」は単なるコスト削減ではありません。マルチプロバイダーのレジリエンス、タスク別の最適化、そして開発の継続性という観点で、単一プロバイダーにはない優位性があります。
