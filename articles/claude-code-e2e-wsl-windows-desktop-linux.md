## はじめに

Windows環境で開発を行っている方の多くは、Linuxベースのツールをシームレスに利用するためにWSL（Windows Subsystem for Linux）を活用していることでしょう。最近話題のAIアシスタントである「Claude Code」などのCLIツールをWindows Desktop環境で動かす際も、WSLを経由して利用することがベストプラクティスの一つです。

しかし、WindowsとLinuxという異なるOSをまたいで開発を行う際、必ず直面するのが**「パス（ファイルの場所を示す文字列）の相互運用性」**の問題です。

本記事では、私が自身の設定リポジトリで実装した「WSLパス変換フック」の設計思想と、実際にE2E（エンドツーエンド）検証を行って判明したWindows/Linux間の相互作用の課題、そしてそれを解決した具体的なステップについて解説します。

## WSL環境におけるパス問題と設計の変遷

WindowsとLinuxでは、ファイルパスの表記方法が根本的に異なります。

- **Windows**: `C:\Users\name\project` （ドライブレター + バックスラッシュ）
- **WSL (Linux)**: `/mnt/c/Users/name/project` （マウントポイント + スラッシュ）

Claude CodeのようなAIエージェントが、Windows側のパスを認識した状態でWSL側のLinuxコマンド（例えば `cd` や `git` など）を実行しようとすると、「そのようなファイルやディレクトリはありません」というエラーになってしまいます。

これを解決するために、最初は特定のマーカー（目印となる文字列）をパスの前後につけて変換する**マーカー方式**を検討しました。しかし、実際にE2E検証（実際の運用に近い状態でのテスト）を行う中で、マーカー方式はAIの出力ぶれによって誤作動を起こすことが判明しました。

そこで、現在は特定のプレフィックス（接頭辞）を持つパスのみを安全に変換対象とする**「ホワイトリストプレフィックス方式」**へと設計を変更し、安定性を高めています。

## Pythonによるパス変換フックの実装

パス変換フックの核心部分は、Pythonを用いて実装しています。設計において最も重視したのは**「可搬性（ポータビリティ）」**です。

後に詳述しますが、ハードコードされたパス（特定のPCのユーザー名などに依存するパス）を排除し、どの環境に持っていってもそのまま動くように設計しています。

以下は、そのコアロジックを簡略化したPythonの実装例です。

```python
import os
import re

def convert_wsl_path(input_path: str) -> str:
    """
    Windows形式のパスをWSL（Linux）形式のパスに変換する
    例: C:\\path\\to\\dir -> /mnt/c/path/to/dir
    """
    if not input_path:
        return input_path

    # 環境変数を展開し、チルダ (~) などのハードコードを避ける
    expanded_path = os.path.expanduser(input_path)

    # Windowsの絶対パス（例: C:\, D:\）をホワイトリストプレフィックスとして検出
    # ドライブレターから始まるパスのみを変換対象とする
    match = re.match(r'^([A-Za-z]):[\\/](.*)', expanded_path)
    if match:
        drive_letter = match.group(1).lower()
        remaining_path = match.group(2).replace('\\', '/')
        
        # WSLのマウントポイントへ組み立て
        wsl_path = f"/mnt/{drive_letter}/{remaining_path}"
        return wsl_path

    # Linuxのパスやその他の形式はそのまま返す
    return expanded_path

# 実行例
if __name__ == "__main__":
    win_path = "D:\\path\\to\\project"
    print(f"Original: {win_path}")
    print(f"Converted: {convert_wsl_path(win_path)}")
    # 出力: /mnt/d/path/to/project
```

このスクリプトは、正規表現を用いて安全にWindowsのドライブパスを検知し、WSLが読み取れる `/mnt/...` 形式へと動的に書き換えます。これをClaude Codeのフック（PreToolUseなどのタイミング）として組み込むことで、AIがエラーに遭遇するのを未然に防ぎます。

## E2E検証で判明した相互作用とポータビリティの向上

設計通りにスクリプトを書いた後、実際の開発フローに組み込んでE2E検証を行いました。その結果、単純な文字列置換のテストでは気付けない「2つの重要な相互作用の課題」が判明しました。

### 1. パス解決の相互作用とハードコードの排除
検証を進める中で、スクリプト内にローカルパス（`/home/username/...` など）を直書きしていると、CI環境や別のPCに持ち出した際にスクリプトがクラッシュすることがわかりました。

これを解決するため、リポジトリ内のすべてのスクリプトと設定ファイルからハードコードされたパスを徹底的に排除しました。Bashスクリプトでは `$HOME` 環境変数を、PowerShellでは `$env:USERPROFILE` を、そしてPythonでは前述のコードにある `os.path.expanduser()` を活用するように統一しています。これにより、新PCに移行してユーザー名が変わっても、スクリプトをコピーするだけで即座に動作するようになりました。

### 2. ファイル実行権限の相互作用
WSL上でWindows側のファイルシステム（`/mnt/c/...` 配下）を操作していると、Gitの管理下にないスクリプトの実行権限（パーミッション）が意図せずリセットされるケースがありました。

例えば、パス変換をテストするための `test_path_rewrite.py` を実行しようとした際、「権限がありません」と弾かれる現象です。WSLとWindowsのファイルシステムは権限の概念が異なるため、相互作用としてこのような問題が起きることがあります。これに対しては、リポジトリのセットアップ手順や同期スクリプトの中で、明示的に実行権限を復元（`chmod +x`）する処理を組み込むことで解決しました。また、動的ロード（`importlib`）を用いている部分には、なぜその手法をとっているのか理由をコメントで明記し、後からの保守性を高めています。

## おわりに

Windows Desktop × WSLという異なるOSが交差する環境は、一度設定してしまえば非常に快適ですが、その間の「パス」や「ファイル権限」のズレは、実運用（E2E）に入って初めて壁として立ちはだかります。

今回実装したWSLパス変換フックとハードコードの排除により、Claude CodeをWindows環境でもストレスなく、かつ堅牢に運用できるようになりました。

異なるOS間のツール連携で躓いている方は、ぜひ「ホワイトリストプレフィックス方式」でのパス変換や、環境変数を活用したポータブルなスクリプト設計を試してみてください。
