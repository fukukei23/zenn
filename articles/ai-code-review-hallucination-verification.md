## はじめに

AIにコードレビューさせていると、**存在しないバグを指摘**することがあります。これが「幻覚（Hallucination）」です。

本記事では、AIコードレビュープロンプトの精度を **v2→v3に改善** した過程を解説します。

## 問題: AIレビューの幻覚

### v2プロンプトの問題

```
AI: 「この関数にはSQLインジェクションの脆弱性があります」
現実: SQLAlchemy ORMを使用しており、パラメータ化クエリ（安全）
```

### 幻覚のパターン

| パターン | 例 |
|---------|-----|
| 過剰検出 | 安全なコードに「脆弱性あり」と指摘 |
| 架空のコード | 存在しない関数・変数を指摘 |
| 一般論の適用 | 「ベストプラクティスとして…」と無関係な指摘 |

## v3の改善: 3LLM合意判定

### アプローチ

同じコードを **3つの異なるLLM** でレビューし、**2つ以上が指摘した項目のみ**を採用:

```python
reviews = {
    "GLM-5.1": review_with(model="glm-5.1", code=target_code),
    "MiniMax": review_with(model="minimax", code=target_code),
    "Claude":  review_with(model="sonnet", code=target_code),
}

# 2つ以上が指摘した項目のみ採用
confirmed = [
    item for item in all_items
    if sum(1 for r in reviews.values() if item in r) >= 2
]
```

### 結果

| 指標 | v2 | v3 |
|------|----|----|
| 幻覚率（架空指摘の割合） | ~30% | ~10% |
| 有益な指摘数 | 5件/レビュー | 4件/レビュー |
| 精度（実バグ/全指摘） | 60% | 85% |

幻覚率は減少、精度は向上。但し**完全にゼロにはならない**。

## プロンプト設計のポイント

### ❌ 悪いプロンプト

```
このコードの問題点を全て指摘してください
```

→ AIが「指摘しなければならない」とプレッシャーを感じ、過剰検出する。

### ✅ 良いプロンプト

```
このコードのレビューをお願いします。
以下の基準で判定してください:
1. 実際のバグ（動作に影響するもの）のみ指摘
2. リファクタリング提案は「改善案」として明示的に分離
3. 確信度（高/中/低）を各指摘に付与
4. 確信度「高」のみを優先して表示
```

→ 「確信度」を要求することで、AIが自己評価し、低確信の指摘を減らす。

## 実運用での発見

### 偽陰性の問題

幻覚を減らすと、逆に**見逃し**が増えるリスク:

```
v2: 10件指摘（3件幻覚、7件実バグ）
v3: 5件指摘（0件幻覚、5件実バグ）← 2件の実バグを見逃し
```

**対策**: v3は「確信度低」も表示し、人間が最終判断する。

### コード規模の影響

| コード行数 | 幻覚率 |
|-----------|--------|
| 50行以下 | 5% |
| 50-200行 | 10% |
| 200行以上 | 25% |

コードが長いほど幻覚が増える。→ **関数単位でレビュー**するのが効果的。

## まとめ

1. **AIレビューの幻覚は避けられない** — 但し軽減は可能
2. **3LLM合意判定で精度向上** — 2/3以上の合意のみ採用
3. **確信度の自己評価が有効** — プロンプトで確信度を要求
4. **短いコード単位でレビュー** — 50行以下が最適
