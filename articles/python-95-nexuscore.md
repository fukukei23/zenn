## はじめに

皆さんは、テストカバレッジ（テストがコードをどれくらい網羅できているかの指標）をどれくらいの数値まで引き上げたことがあるでしょうか？

「正常系のテストは書けるけれど、エラー発生時の処理や例外ハンドラなど、いわゆる『まず動かない経路』のテストがうまく書けない……」という悩みを持つ開発者は少なくありません。

私が開発に参加しているオープンソースプロジェクト「**NexusCore**」でも、かつてはそんな課題を抱えていました。しかし最近、認証・例外ハンドラ・DBフェイルオーバーなどの「フォールバック経路（代替処理）」に焦点を当ててテストを大幅に改善し、一部のモジュールで**カバレッジ95%超え**を達成しました。

本記事では、NexusCoreで実際に行ったテスト改善のアプローチをベースに、`MagicMock` / `AsyncMock` を使ったエラー経路のテスト技法や、カバレッジ計測ツールの結果の読み方について解説します。初心者の方でも「エラー処理のテストってどう書けばいいんだろう？」と感じている方に向けて、実務レベルの具体的なコツを紹介します。

## 「フォールバック経路」とは？なぜテストが難しいのか

WebアプリケーションやAPIの開発において、「フォールバック（Fallback）経路」とは、メインの処理が失敗した際に実行される**代替の処理やエラーからの復旧処理**を指します。

たとえば以下のようなケースです。
- メインのDBがダウンしているときに、キャッシュや環境変数からデータを読み込む
- JWT認証モジュールのインポートに失敗した（`ImportError`）際に、別の認証方式に切り替える
- APIリクエストの処理中に予期せぬ例外が発生した際、ロールバックを行ってからユーザーにエラーレスポンスを返す

これらは普段の開発や手動テストでは「まず発生しない（動かない）経路」であるため、テストを書き忘れがちです。しかし、いざという時にシステムがクラッシュしないための重要な防波堤となります。

## Mockを使って「わざとエラーを起こす」テスト技法

「まず動かない経路」をテストするには、**意図的にエラーを発生させる**必要があります。ここで威力を発揮するのが、Pythonの標準ライブラリ `unittest.mock` にある `MagicMock` や `patch`、非同期処理用の `AsyncMock` です。

NexusCoreのPR（Issue #90: api_keys.py の改善）で実際に行った、DBエラー（`SQLAlchemyError`）発生時のテストコード例を見てみましょう。

### 具体的なコード例：DB例外時のロールバックテスト

以下のように、`patch` デコレータを使ってDBのセッション操作を擬似的にエラー（`side_effect`）に設定します。

```python
import pytest
from unittest.mock import MagicMock, patch
from sqlalchemy.exc import SQLAlchemyError
from fastapi import HTTPException

# テスト対象の関数が存在するモジュールを指定
@patch("src.nexuscore.api.routes.api_keys.get_db")
def test_api_key_sqlalchemy_error_rollback(mock_get_db):
    """
    APIキー作成時にDBエラーが発生した場合、
    正しくロールバックされ、HTTPExceptionが返るかをテストする
    """
    # 1. モックのDBセッションを用意する
    mock_db = MagicMock()
    mock_get_db.return_value = mock_db
    
    # 2. DBのコミット処理で意図的にSQLAlchemyErrorを発生させる
    mock_db.commit.side_effect = SQLAlchemyError("DB Connection Lost")

    # 3. テスト対象の関数を実行し、例外が発生するか検証
    with pytest.raises(HTTPException) as exc_info:
        create_api_key(user_id=123, db=mock_db)
    
    # 4. ステータスコードが500であること
    assert exc_info.value.status_code == 500
    
    # 5. エラー発生時のロールバック（取り消し処理）が確実に呼ばれたか検証
    mock_db.rollback.assert_called_once()
```

### 押さえておくべきポイント

1. **`side_effect` の活用**
   `MagicMock` の `side_effect` に例外クラスや例外インスタンスを設定すると、そのモック関数が呼ばれた瞬間に擬似的にエラーを発生させることができます。これにより、滅多に起こらないDB障害を何度でも再現できます。

2. **`assert_called_once` での事後確認**
   エラーが起きたことだけでなく、「エラー起因でロールバック処理がちゃんと実行されたか」を `mock_db.rollback.assert_called_once()` で検証します。これにより、例外ハンドリングの堅牢性を担保します。

3. **非同期処理（AsyncMock）への拡張**
   もしテスト対象が `async def` で定義された非同期関数であれば、`unittest.mock.AsyncMock` を使用します。基本的な考え方は同じで、非同期のDBクライアントや外部API呼び出しのモック化に非常に有効です。NexusCore（Issue #94）では、この技術を使って非同期のJWT認証フォールバックを検証しました。

## カバレッジレポートの読み方と回帰テストの重要性

テストを書く際、「どこまで書けば95%に届くのか」を知るにはツールの力が欠かせません。Pythonでは `pytest-cov` を使ってカバレッジを計測します。

### カバレッジレポートで「未到達の行」を見つける

ターミナルで以下のコマンドを実行してみてください。

```bash
pytest --cov=src/nexuscore/api --cov-report=term-missing
```

`--cov-report=term-missing` オプションをつけることで、**テストが通っていない（カバレッジ対象外の）具体的な行番号**が表示されます。

NexusCoreのPR（Issue #96）では、レポートに表示された `L51-53`（51行目〜53行目）などの未カバー箇所を確認し、「ここはDBが初期化されていない場合のフォールバック処理だな」と特定した上で、先ほど紹介したようなMock技法のテストを追加していきました。

### 回帰テスト（デグレ確認）の徹底

新しいテストを追加してカバレッジが上がっても、**既存のテストが壊れていては意味がありません**。
NexusCoreでは、新規テストを追加した後に必ずフルスイートのテストを実行して回帰（デグレード）がないかを確認しています。

実際にPRには「フルスイート 5400+ passed, 0 failed（回帰なし）」という記録が残っています。カバレッジ数字だけを追うのではなく、既存機能を破壊していないことを確認するプロセスこそが、安全なリファクタリングとテスト改善の鍵です。

## おわりに

テストカバレッジ95%という数字は、決して自己満足の数字ではありません。「普段は絶対に通らないエラー経路」までテストを書くことで、本番環境での突然の障害に対する強い安心感を得ることができます。

- **`patch` と `MagicMock` / `AsyncMock`** を使って外部依存（DB・認証など）を擬似的にエラーにする
- **`pytest-cov` のレポート**で未到達の行を特定する
- **カバレッジ数字だけでなく、既存テストの回帰なし**を確認する

この3つのステップを意識するだけで、あなたのプロジェクトのテストの品質は劇的に向上するはずです。「エラー処理のテストが書けない」という方は、ぜひこの記事で紹介したMock技法を参考に、まずは1つ「わざとエラーを起こすテスト」から書き始めてみてください！
