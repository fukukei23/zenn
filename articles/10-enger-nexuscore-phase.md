## はじめに

ソフトウェア開発の現場において、「最初はシンプルだったはずのコードが、気がつけば200行を超える長関数（God Method）になっている」という経験はありませんか？

条件分岐がネストし、変数があちこちで書き換えられ、手を入れようとするだけでバグを引き起こしそうな闇のコード。今回は、私が携わるPythonプロジェクト「**NexusCore**」で実際に行った、200行超えの長関数を10以上のファイルに分割したリファクタリングの実例を紹介します。

本記事では、抽象的なリファクタリングの理論ではなく、「文字列組み立て」「APIルーティング」「例外処理」といった具体的なパターン別に、NexusCoreのPhase 1〜4のコミット履歴に基づきながら、実践的な長関数攻略法を解説します。

## Phase 1 & 2 — 「長すぎる関数」をパターン別に切り裂く

長関数の分割で最初にやるべきことは、「**関数の中で行われている処理の役割**」に注目して切り出すことです。NexusCoreでは、特に以下の2つのパターンが顕著でした。

- 文字列組み立て（Phase 1）
- API・ルーティング処理（Phase 2）

### 文字列組み立て関数の分割（199行 → 57行）

NexusCoreでは、AIエージェントに渡すプロンプトやSQLクエリを動的に組み立てる関数が巨大化していました。元々199行あった関数は、「データの取得」「条件による文字列の追加」「フォーマットの適用」といった処理が1つの関数に混在していました。

これを「データの取得」「フォーマットの適用」「文字列の結合」といった単位でヘルパー関数として切り出すことで、メインの関数は57行までスリム化できました。

```python
# Before: すべてが1つの巨大な関数に詰め込まれた長関数（イメージ）
def build_complex_query(context, user_input, options):
    query = ""
    if options.get("include_history"):
        # 過去履歴を取得して文字列に追加する処理 (20行くらい)
        pass
    if options.get("format_json"):
        # JSONフォーマットに変換する処理 (30行くらい)
        pass
    # その他いろんな処理 (150行くらい)
    return query

# After: 役割ごとに小さな関数に分割する
def get_history_text(context):
    """過去履歴の取得とフォーマット"""
    # ループや条件分岐をここに隔離
    return formatted_history

def format_payload(user_input, options):
    """データのフォーマット"""
    # 複雑なフォーマット処理をここに隔離
    return payload

def build_complex_query(context, user_input, options):
    # メインの関数は「組み立て」だけに専念する
    history = get_history_text(context)
    payload = format_payload(user_input, options)
    
    query = f"{history}\n{payload}"
    return query
```

### API・ルーティング処理の分割（159行 → 44行）

Phase 2では、APIのエンドポイント処理に焦点を当てました。ルーティングの定義、認証、ビジネスロジックが1つの関数に詰め込まれていると、可読性が著しく低下します。

NexusCoreでは、APIのルーティング処理を「依存関係の注入」「バリデーション」「メインロジック」という層に分割しました。これにより、最大159行あったAPI関数を44行まで削減し、他の6つのAPI関数も同様に30〜70行程度にスリム化することができました。

## Phase 3 — 「ベアエクセプト（bare except）」を全廃し、例外処理を整理する

長関数が生まれるもう一つの大きな原因に「雑な例外処理」があります。例外をキャッチしてはその場で長々とエラーハンドリングやフォールバック処理を書くと、どうしても関数が肥大化します。

NexusCoreの「Phase 3」では、Pythonでアンチパターンとされる`except:`（ベアエクセプト）や`except Exception:`の全廃に挑みました。実に100ファイル以上に及ぶ修正でしたが、例外処理の切り出しによってメインロジックの見通しが劇的に良くなりました。

```python
# Before: 何でもかんでもキャッチして長くなる
def process_data(data):
    try:
        # メインの処理
        pass
    except:
        # ログ出力、リトライ、フォールバックなどがダラダラと続く
        pass

# After: 例外ごとのハンドラを分離する
def handle_api_error(e: APIError):
    """APIエラー時の専用ハンドラ"""
    # エラー時の複雑な処理をここに隔離
    pass

def process_data(data):
    try:
        return execute_data_process(data)
    except APIError as e:
        return handle_api_error(e)
    except ValidationError as e:
        logging.error(f"Validation failed: {e}")
        raise
```

例外処理を関数の外（または専用のハンドラ）に切り出すことで、「この関数が本来何をしたいのか」というコアロジックが一瞬で把握できるようになります。

## Phase 4 — 最後の仕上げ（クリーンアップとテスト）

関数の分割が一通り終わったら、必ずやっておきたいのがクリーンアップです。NexusCoreのPhase 4では、主に以下の2つの作業を行いました。

1. **未使用importの削除**: 関数を別ファイルに移動したことで、元のファイルに不要なimportが残っていたため、6ファイル分を一括で整理しました。
2. **ハードコードの定数化**: 3ファイルに散らばっていたハードコードされたURLや固定文字列を定数としてファイルの先頭にまとめました。

そして、リファクタリングにおいて最も重要なのが**「テスト」**です。NexusCoreのPR（Pull Request）履歴を見ると、リファクタリングと並行して「テストカバレッジを82%から95%以上に向上させる」といった地道な作業が行われています。

長関数を小さな関数に分割する最大のメリットは、**モックが使いやすくなり、結果的にテストが圧倒的に書きやすくなること**です。巨大な関数をテストしようとすると、内部の状態を再現するだけで大変ですが、分割後の小さな関数なら、入力と出力だけを気にしてテストを書くことができます。

## おわりに

200行を超える長関数は、一見すると「1つの関数で全て完結しているから分かりやすい」と錯覚しがちですが、実際は変更に弱く、テストも困難な技術的負債です。

NexusCoreの実例で見たように、いきなり全てを書き換えるのではなく、以下のようにフェーズを分けて進めるのが確実なアプローチです。

1. **Phase 1 & 2**: 文字列組み立てやAPIルーティングなど、処理の塊ごとに切り出す
2. **Phase 3**: 雑な例外処理（bare except）を適切に分離する
3. **Phase 4**: 不要なimportの削除・定数化などのクリーンアップとテストの強化

長関数の解消は、コードの可読性を高めるだけでなく、プロジェクト全体の保守性を大きく向上させます。「長関数は敵」と割り切って、まずは1つの役割を持つ関数を切り出すところから始めてみてはいかがでしょうか。
