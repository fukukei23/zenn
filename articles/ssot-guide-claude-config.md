## はじめに

開発を進めていると、必ず直面するのが「タスクの迷子問題」です。
「あれ、前に調査していた未解決のバグ、どこにメモしたっけ？」「Issueをクローズしたけど、関連する残タスクはどうなった？」

このようなタスクの迷子は、小さなプロジェクトであっても積み重なると致命的な生産性の低下を招きます。今回は、私が自身のプロジェクトである `claude-config` リポジトリ（※ssot-guideというガイドラインに基づく設定管理用リポジトリ）の「フェーズ3.6」で実装した、**未解決→バックログ追加確認ゲート**という仕組みをご紹介します。

Pythonを使った半自動化の仕組みを、初心者の方にもわかりやすいように具体的なコードとともに解説します。

## タスクが迷子になる原因と設計思想

なぜタスクは迷子になってしまうのでしょうか。実務経験上、大きな理由は以下の2つです。

1. **一元管理（SSOT）の欠如**: メモ、Issue、ローカルのテキストファイルなど、情報があちこちに散らばっている。
2. **仕分けのタイミング逃し**: 未解決のタスクを見つけたとき、「あとでやる」と考えてその場で処理（あるいは無視）してしまい、バックログ（今後やることのリスト）に記録し忘れる。

これを防ぐためには、情報の源泉を一つにする（SSOT: Single Source of Truth）とともに、**「未解決のタスクを見つけた瞬間に、バックログへ追加するか確認を促すゲート（関所）」**を設ける必要があります。

完全に自動でバックログに突っ込むと、今度はノイズだらけになってしまいます。そのため、「検出は自動、判断は人間」の**半自動**という設計にしました。

## Pythonで実装する「バックログ追加確認ゲート」

実際にどのようにゲートを実装したか、Pythonのコード例を見てみましょう。
ここでは、GitHub上の未解決Issue（またはローカルのタスクファイル）をスキャンし、対話形式でバックログへの追加を促す簡易的なスクリプトを紹介します。

```python
import json
import subprocess
from pathlib import Path

def fetch_unresolved_items():
    """
    未解決のタスク（Issueなど）を取得する関数。
    実際のプロジェクトではGitHub CLI (gh) などを呼び出して取得します。
    """
    # 例: gh issue list --state open --json number,title
    try:
        # ここではダミーデータとして返します
        return [
            {"id": 101, "title": "設定ファイルのパスが環境依存しているバグを修正"},
            {"id": 102, "title": "新機能のAPIドキュメントを追加"}
        ]
    except Exception as e:
        print(f"タスクの取得に失敗しました: {e}")
        return []

def ask_to_add_backlog(item_title: str) -> bool:
    """ユーザーにバックログへの追加を尋ねるゲート（半自動の判断部分）"""
    print(f"\n[未解決アイテム発見] {item_title}")
    response = input("このアイテムをバックログに追加しますか？ (y/n): ")
    return response.lower() == 'y'

def main():
    # /path/to/your/project/ のような汎用的なパスを指定します
    backlog_file = Path("/path/to/your/project/BACKLOG.md")
    
    # ファイルが存在しない場合は作成
    backlog_file.parent.mkdir(parents=True, exist_ok=True)
    
    items = fetch_unresolved_items()
    if not items:
        print("未解決のアイテムはありません。素晴らしいです！")
        return

    added_items = []
    
    for item in items:
        if ask_to_add_backlog(item["title"]):
            added_items.append(f"- [ ] {item['title']} (ID: {item['id']})")
    
    # バックログへの書き込み処理
    if added_items:
        with backlog_file.open("a", encoding="utf-8") as f:
            f.write("\n## 追加されたバックログ\n")
            f.write("\n".join(added_items) + "\n")
        print(f"\n{len(added_items)}件のタスクをバックログに追加しました！")
    else:
        print("\n追加されたタスクはありません。")

if __name__ == "__main__":
    main()
```

このスクリプトのポイントは、`ask_to_add_backlog` 関数です。人間が `y` または `n` を入力することで、ノイズになるタスクを弾きつつ、本当に必要なタスクだけを `BACKLOG.md` （Single Source of Truth）に強制的に書き込ませています。

## プロジェクトライフサイクルにゲートを組み込む

このゲートをただのスクリプトとして置いておくだけでは、結局「実行し忘る」という迷子問題を引き起こします。そこで、開発のライフサイクル（セッション）の特定のタイミングで、このゲートが自動的に作動するようにフックします。

私のリポジトリでは以下の3つのレイヤー（ABC層）を意識して組み込んでいます。

1. **A層（セッション開始時）**: 前回の未解決タスクをスキャンして一覧を表示。
2. **B層（タスク選択・終了時）**: 作業が終わったとき、未解決のまま残ったアイテムがないか確認し、あればバックログ追加ゲートを起動。
3. **C層（セッション再開時）**: 機械的にローカルファイルをスキャンし、迷子になりそうなタスクを検知。

例えば、シェルスクリプト（bash）で開発終了時のフックを作り、バックグラウンドで通知を飛ばしつつ上記のPythonスクリプトを叩くような構成にすると、作業の流れを邪魔することなくタスクの回収が可能です。

## おわりに

「タスクが迷子になる」のは人間の記憶力の問題ではなく、**システム（仕組み）の欠如**です。

- SSOT（情報の唯一の源泉）を `BACKLOG.md` のような一つのファイルに集約する
- 未解決アイテムを半自動で検知し、人間の判断でバックログに振り分けるゲートを作る

この2つを導入するだけでも、開発体験は劇的に改善されます。「どこにメモしたっけ？」と過去のチャットログを漁る時間を減らし、本質的なコーディングに時間を割けるようになります。ぜひ皆さんのプロジェクトでも、お手製の「確認ゲート」を設けてみてはいかがでしょうか。
