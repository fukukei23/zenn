## はじめに

WindowsとLinuxをシームレスに扱えるWSL2は非常に便利ですが、新PCへの乗り換えや初期設定のし直しで「突然スクリプトが動かなくなる」という苦い経験をしたことはないでしょうか。

私は最近、WindowsとWSL2を組み合わせた開発環境の整備を行っていたのですが、新しくPCを移行した際、WSL2のユーザー名を変更したせいで多数のスクリプトが一斉にエラーを吐いて停止してしまうという自体に直面しました。

原因は、スクリプトや設定ファイルの中に**ユーザー名を直書き（ハードコード）**していたことでした。本記事では、具体的なPythonやBashのスクリプトを例に挙げ、環境に依存しない「スクリプト可搬性」の高い設計パターンを初心者向けに解説します。

## ハードコードされたパスが引き起こす悲劇

開発を進めていると、ログの出力先や設定ファイルの読み込み先など、どうしても絶対パスを指定したい場面が出てきます。

例えば、以下のようにパスを直接書き込んでいたとします。

```bash
# NG例: ユーザー名が直接パスに含まれている
SCRIPT_PATH="/home/taro/scripts/notify.sh"
```

この環境では問題なく動作しますが、新PCに移行してWSL2のユーザー名が「jiro」になった瞬間、このパスは存在しない場所を指し示すようになり、スクリプトは「ファイルが見つかりません（No such file or directory）」というエラーを引き起こします。

複数人で開発を行う場合でも、それぞれのローカル環境でユーザー名が異なるため、Gitなどでコードを共有する際に毎回パスを書き直す手間が発生してしまいます。

## $HOME変数とPythonのexpanduser()で解決

この問題を解決する一番簡単で効果的な方法は、OSが標準で提供している**環境変数**を活用することです。

### Bashスクリプトの場合

Linux（WSL2）のBashスクリプトでは、`$HOME` という環境変数が現在のユーザーのホームディレクトリ（例: `/home/taro`）を自動的に保持しています。これを利用して、先ほどのパスを次のように書き換えます。

```bash
# OK例: $HOME変数を使って動的にパスを解決
SCRIPT_PATH="$HOME/scripts/notify.sh"
```

こうするだけで、ユーザー名が何であっても正しいパスを参照できるようになります。

### Pythonスクリプトの場合

Pythonでファイルのパスを操作する際は、標準ライブラリの `pathlib` を使うのがモダンな書き方です。自分のリポジトリ内の設定読み込み処理でも、以下のように改善を行いました。

`Path` オブジェクトに `~`（チルダ）を含め、`expanduser()` メソッドを呼び出すことで、実行環境のホームディレクトリを自動的に展開してくれます。

```python
from pathlib import Path

# NG例
# log_dir = Path("/home/taro/.claude/logs")

# OK例: expanduser()でチルダを現在のユーザーのパスに展開
log_dir = Path("~/.claude/logs").expanduser()

# ディレクトリが存在しない場合は作成する
log_dir.mkdir(parents=True, exist_ok=True)

print(f"ログディレクトリ: {log_dir}")
```

このように `expanduser()` を挟むだけで、Windows環境であってもLinux環境であっても、実行したユーザーの適切なホームディレクトリにログフォルダが作成されるようになります。

## 設定ファイル（JSON）とPowerShellの連携

スクリプト本体だけでなく、JSONなどの設定ファイルにも工夫を凝らすことで、さらに堅牢な構成を作ることができます。

### JSON設定ファイルでの工夫

Pythonスクリプトから読み込むJSONファイル内にも、直接パスを書きたい場合があります。その場合は、直接ユーザー名を指定するのではなく、チルダ（`~`）を使っておきます。

```json
{
  "logging": {
    "dir": "~/.claude/logs"
  }
}
```

この設定ファイルを読み込むPythonスクリプト側で、先ほどの `expanduser()` を使ってパスを展開すれば、設定ファイル自体もどの環境でもそのまま使い回すことができます。

### Windows側（PowerShell）でのパス解決

WindowsとWSL2を連携させている場合、Windows側からスクリプトを叩くためのPowerShellスクリプトを書くこともあるでしょう。その場合は `$env:USERPROFILE` という環境変数を併用します。

```powershell
# OK例: Windowsのユーザーディレクトリを動的に取得
$ConfigDir = "$env:USERPROFILE\.config\my-app"
Write-Host "設定ディレクトリ: $ConfigDir"
```

WSL2側は `$HOME`、Windows側は `$env:USERPROFILE` と使い分けることで、どちらの環境に移行しても壊れない「スクリプトの可搬性」を高めることができます。

## おわりに

今回紹介したような「ハードコードを避け、環境変数を利用する」という手法は、慣れてしまえば当たり前のことですが、開発が佳境に入っているとうっかり見落としがちな罠でもあります。

新PCへの移行時や、チームメンバーへの環境共有時に「スクリプトをコピーするだけでサクッと動く」状態を作っておくと、心理的な負担が劇的に下がります。

これからWindows + WSL2での開発環境を構築するという方は、ぜひ普段のスクリプト作成から `$HOME` や `expanduser()` などの活用を意識してみてください。小さな工夫が、後の大きなトラブルを防いでくれます。
