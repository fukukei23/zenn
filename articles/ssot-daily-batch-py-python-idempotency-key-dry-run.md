# はじめに

日々の開発業務やドキュメント管理を自動化する「デイリーバッチ」を運用していると、必ず「再実行が怖い」という壁にぶつかります。

- 「途中でネットワークエラーで止まってしまったけど、再実行して二重にデータが登録されたらどうしよう」
- 「テスト中に間違えて本番用のバッチを走らせて、不要なファイルが大量に更新されてしまった...」

こうした「多重実行」や「誤実行」の恐怖を払拭するためには、バッチ処理に**べき等性（Idempotency）**や**安全装置**を組み込む必要があります。

本記事では、私が自身のリポジトリで構築した「SSOT（Single Source of Truth：信頼できる唯一の情報源）」の日常運用を自動化するバッチ基盤の実装例を解説します。具体的には、以下の要素を実装したPythonスクリプトとシェルスクリプトの連携について紹介します。

- 重複実行を防ぐ `idempotency_key`
- エラー時の賢いリトライ `retry_meaning_with_backoff`
- ステップ依存ルールと `dry-run` モードによる安全な実行

# 1. 重複実行を防ぐ「べき等性」の実現

バッチ処理において最も重要な概念の一つが「べき等性」です。これは**「同じ操作を何度繰り返しても、結果が同じになる」という性質**を指します。

外部APIを呼び出してデータを更新する処理や、DBのレコードを作成する処理などでは、べき等性がないとバッチを再実行するたびに同じデータが量産されてしまいます。そこで導入するのが `idempotency_key`（べき等キー）です。

処理の対象となるデータ（例：ファイル名やタスクIDなど）から一意のハッシュ値を生成し、それが「すでに処理済みであるか」を記録・参照します。

バッチを再実行した際、すでに処理が完了している `idempotency_key` にヒットした場合は、対象の処理を安全にスキップします。これにより、「途中で止まってしまったから最初からやり直し」という非効率な運用から卒業し、何度実行しても最終的なシステムの状態が同じになる堅牢な基盤を作ることができます。

# 2. 外部APIの呼び出しと賢いリトライ（Backoff）

デイリーバッチでは、LLM（大規模言語モデル）などの外部APIを呼び出すことがよくあります。しかし、外部APIは時に `429 Too Many Requests`（アクセス過多）や `503 Service Unavailable`（一時的なサーバーエラー）を返すことがあります。

このような一時的なエラーでバッチが即座に異常終了してしまうと、運用者の負担が大きくなります。そこで、HTTPステータスコードに応じて自動的に待機時間を延ばしながらリトライする `retry_meaning_with_backoff` という仕組みを実装しました。

以下は、その概念をシンプルに実装したPythonのサンプルコードです。

```python
import time
import requests

def retry_meaning_with_backoff(func, max_retries=3):
    """
    ステータスコードに応じてバックオフ（指数的待機）を行うリトライ関数
    """
    for attempt in range(max_retries):
        try:
            response = func()
            
            # 2xx系の成功
            if 200 <= response.status_code < 300:
                return response
            
            # リトライすべきステータスコード（429, 503など）
            if response.status_code in [429, 503]:
                wait_time = 2 ** attempt  # 指数バックオフ (1秒, 2秒, 4秒...)
                print(f"サーバー混雑中。{wait_time}秒待機してリトライします...")
                time.sleep(wait_time)
                continue
                
            # それ以外の4xxエラー（認証エラーやBad Requestなど）は即時終了
            response.raise_for_status()
            
        except requests.exceptions.RequestException as e:
            # ネットワーク切断などの場合もリトライ
            if attempt < max_retries - 1:
                time.sleep(2 ** attempt)
            else:
                raise e
                
    raise Exception("最大リトライ回数に達しましたが、処理が成功しませんでした。")
```

このように実装することで、サーバーが一時的にダウンしていても自動的に復旧を待ち、無駄なAPI呼び出しを防ぎながら確実に処理を進めることができます。

# 3. ステップ依存と dry-run による安全なオーケストレーション

バッチが複数の処理から構成される場合、「ステップAが成功したらステップBを実行する」という依存関係を管理するオーケストレータが必要です。

Python側のメインスクリプトでは、各ステップの成否を集計し、途中で致命的なエラーが発生した場合は後続ステップを中止する「ステップ依存ルール」を定義しています。

さらに、誤って意図しない変更が加わるのを防ぐために `dry-run`（ドライラン）モードを実装しています。これを有効にすると、実際のファイル書き換えやAPI呼び出し、Gitのコミットを行わずに「どのファイルが更新される予定か」「どのような処理が走るか」というシミュレーション結果だけを出力します。

```python
def run_batch(dry_run=False):
    """
    バッチのメイン実行関数（オーケストレータ）
    """
    print("=== デイリーバッチ 開始 ===")
    if dry_run:
        print("※ DRY-RUNモードが有効です。実際の変更は行われません。")
    
    # ステップ1: インデックスの更新
    step1_success = update_indexes(dry_run=dry_run)
    
    # ステップ1が失敗した場合は、後続ステップを中止する
    if not step1_success:
        print("ステップ1が失敗したため、後続処理を中止します。")
        return False
    
    # ステップ2: 保留中タスクの処理（API通信を含む）
    step2_success = process_pending_tasks(dry_run=dry_run)
    
    # 結果の集計
    summarize_results(step1_success, step2_success)
    print("=== デイリーバッチ 終了 ===")
    return True
```

### シェルスクリプトによる多重起動防止と安全終了

Pythonスクリプトを実際にcronなどで定期実行する際には、さらにシステム側の安全装置が必要です。バッチの起動を管理するシェルスクリプトでは、以下のような工夫をしています。

- **stampファイルの利用**: バッチ実行中であることを示すファイルを作成し、多重起動を防止します。
- **trapコマンドによる安全終了**: バッチが予期せぬエラーや `Ctrl+C` で強制終了された際に、確実にstampファイルを削除します。これにより、次回のバッチ実行がブロックされるのを防ぎます。

# おわりに

「再実行が怖い」バッチは、本来自動化が目指す「運用負荷の軽減」のメリットを半減させてしまいます。しかし、以下のような仕組みを組み合わせることで、誰もが安心して実行できる堅牢なパイプラインを構築できます。

1. **べき等性の確保** により、重複実行を気にしなくてよくなる
2. **バックオフリトライ** により、一時的なネットワーク障害を自動で吸収できる
3. **dry-run と trap** により、開発時のテストやすぐさま終了時の安全性が担保される

今回紹介した実装は、日々のタスクを自動化する際のベストプラクティスの一部です。これらを導入することで、夜間の自動実行や障害時のリカバリが格段にやりやすくなります。安全なバッチ基盤を作って、より価値のある開発タスクに時間を割けるようにしていきましょう！
