# はじめに

個人開発やチーム開発を問わず、CI（継続的インテグレーション）の画面が「赤」で埋め尽くされていると、開発のモチベーションが下がってしまいますよね。

私が現在開発しているプロジェクト（※以下、自作プロジェクト）でも、ある時期にCI上で **44件のテストエラー（failed）** が発生していることに気づきました。しかし、焦って一度に全てを直そうとするのはかえって火消し作業を長引かせる原因になります。

この記事では、44件あったCIのテスト失敗を段階的に「0（完全緑化）」にまで落とし込んだ具体的な戦い方を記録します。特に、大量のエラーに直面して途方に暮れている初心者の方に向けて、**エラーのパターン分類**や**CIの「嘘緑」を検知する方法**など、実務で使えるワークフローをわかりやすく解説します。

---

## 1. CIの「嘘緑」を検知し、依存関係を整理する

いきなりテストコードを直し始める前に、まずは**「なぜCIが通らないのか（あるいは嘘の成功を報告しているのか）」**という根本原因を整理します。

今回のプロジェクトでは、過去に「CI嘘緑」と呼ばれる現象が起きていました。これは、CI上でテスト自体が正しく実行されていないのに、エラーを握りつぶして成功（緑色）としてしまっている状態です。

これを解決するために、まずは以下の2点を行いました。

1. **依存関係の漏れを修正**
   `httpx` や `pydantic` など、CI環境にインストールされていないパッケージが原因でテストが落ちていたため、`requirements.txt` や `pyproject.toml` に依存関係を追加しました。
2. **`Makefile` の整備によるローカル同期**
   開発者のローカル環境とCI環境のズレを防ぐため、`make test` コマンドを実行する前に必ず `make install`（依存ライブラリのインストール）が走るように変更しました。

```makefile:Makefileの改善例
.PHONY: install test

install:
	@echo "Installing dependencies..."
	pip install -r requirements.txt

test: install  # testの前に必ずinstallを依存させる
	@echo "Running tests..."
	pytest
```

このような「環境の整備」を先に行うことで、真のテスト結果がCI上に表示されるようになります。

---

## 2. エラーをパターン分けして「少しずつ」解消する

環境が整備できたところで、いよいよ44件のエラーと向き合います。ここで大切なのが**「エラーの分類」**です。

44件のエラーを詳細に分析したところ、そのうち **39件が同じ原因（PriceIntegrityError系）** であることが判明しました。残りの5件は別原因（依存漏れやフォーマット崩れなど）だったため、まずは別原因の5件を先に解消し、残りの39件に集中して取り組みました。

### `pytest -k` を使った部分実行

39件のエラーが `PriceIntegrityError` という同じ例外で止まっていたため、これをモジュールごとに細分化して直していきました。

```bash
# 例: calculatorに関するテストだけを抽出して実行
pytest -k "test_calculator"
```

### Pythonコードの修正例

エラーの原因は、DBのマイグレーション（スキーマ定義）が古く、テストコードで要求している新しいカラム（`purchase_price_source` など）が存在していなかったことです。

そのため、AlembicなどのマイグレーションツールでDBスキーマを更新し、テストコード側も最新の仕様に合わせて修正しました。

```python:テストコードの修正イメージ
def test_calc_profit_basic():
    # 修正前: 存在しないカラムを参照してPriceIntegrityErrorが発生していた
    # product = get_product(id=1)
    
    # 修正後: マイグレーションで追加されたカラムを明示的に設定
    product = Product(
        id=1,
        purchase_price=1000,
        purchase_price_source="manual", # 追加されたカラムを補完
        selling_price=1500,
        selling_price_source="auto"     # 追加されたカラムを補完
    )
    
    profit = calc_profit(product)
    assert profit == 500
```

コミットメッセージには「`PriceIntegrityError系残り26件のうち test_calculator 17件を解消`」のように、**「あといくつ残っているか」**を必ず記載するようにしました。これにより、進捗が可視化され、途中で挫折せずに最後の1件まで根絶できました。

---

## 3. 完全緑化を維持するためのワークフロー

すべてのテストを「0 failed」にすることができたら、次は**この状態を維持すること**が重要です。今回は以下のワークフローを導入しました。

### ① READMEにCIバッジを貼る
GitHubのREADMEにCIのステータスバッジを貼り、トップページで常に「緑」か「赤」かがわかるようにしました。これにより、「赤い状態のまま放置する」という事態を心理的に防げます。

### ② テスト数とカバレッジを正確にする
以前はREADMEに「テスト数2070、カバレッジ78.5%」と適当な数字が書かれていましたが、これを実際の数字（2155 passedなど）に修正し、CIが完全緑化したタイミングで改めてカバレッジを正確に再計測しました。

### ③ RuffなどによるLintの自動化
コードスタイルのズレによるエラーを防ぐため、`Ruff` などのLinterを導入し、未使用のimport削除やフォーマット整形をCI上で自動実行するようにしました。

# おわりに

44件のテスト失敗を0にするまでの道のりは、一見遠回りに感じるかもしれません。しかし、以下のようなステップを踏むことで、確実に「完全緑化」を達成できます。

1. **環境の依存関係を整理し、ローカルとCIのズレをなくす**
2. **大量のエラーをパターン分けし、性質ごとに分割して修正する**
3. **進捗をコミットメッセージに残し、ステータスバッジで可視化する**

CIが完全に緑色になった時の安心感と開発スピードの向上は、それまでの苦労を凌駕するほどの価値があります。テストのエラーに押しつぶされそうになっている方は、ぜひ「エラーの分類」から始めてみてください。

この記事が、同じようにテストエラーの修正に立ち向かう方の参考になれば幸いです。
