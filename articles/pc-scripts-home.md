# はじめに

パソコンを買い替えたり、開発環境をWindowsからMac、あるいはWSL（Windows Subsystem for Linux）へ変更した際、「あれ？昨日まで動いていたスクリプトが急にエラーになる！」と焦った経験はありませんか？

その原因の多くは、コードの中に**「特定のPCでしか通用しないユーザー名などのパスが直書きされていること」**にあります。

今回は、私が自身の開発環境向けの設定ファイルやスクリプトを管理しているリポジトリで行った修正を例に、新PCに移行しても「スクリプトをコピーするだけでそのまま動く」仕組み（可搬性の向上）を初心者向けに解説します。

# パスの「ハードコード」が引き起こす罠

プログラムを書いていると、設定ファイルの場所やログの出力先などを指定する必要があります。例えば、以下のように直接パスを書き込んでしまうことを「ハードコード（直書き）」と呼びます。

```python
# 悪い例：特定のユーザー名がパスに含まれている
log_file_path = "/home/taro/my_project/logs/app.log"
```

このコードは、ユーザー名が `taro` であるPCでは完璧に動作します。しかし、新しいPCでユーザー名を `hanako` にした途端、`/home/taro/` というディレクトリが存在しないため「ファイルが見つかりません」というエラーになってしまいます。

自分専用のスクリプトであれば「該当箇所を検索して一括置換すればいい」と思うかもしれません。しかし、設定ファイルや連携用のスクリプトが数十個に及ぶと、置換作業は非常に手間がかかり、修正し忘れによるバグの温床になります。

これを防ぐためには、**「現在ログインしているユーザーのホームディレクトリ」を自動的に取得する仕組み**を使う必要があります。

# 環境変数を使ってパスを一般化しよう

OSには、ユーザーの環境情報を保持している「環境変数」という仕組みがあります。この中にある `$HOME`（Windows PowerShellの場合は `$env:USERPROFILE`）という変数を使うことで、実行環境のユーザー名が何であっても正しいパスへたどり着くことができます。

先日私が管理するリポジトリの設定変更（Pull Request）でも、すべてのスクリプト・設定ファイルから直書きのユーザー名を除去し、以下のように各言語の機能を使って書き換えました。

## 1. Pythonスクリプトの場合

Pythonでホームディレクトリのパスを取得するには、`~`（チルダ）と標準ライブラリである `pathlib` の `expanduser()` メソッドを活用するのが最もモダンで安全な方法です。

```python
from pathlib import Path

# ハードコードされたパスの代わりに ~/ を使って相対的に指定する
config_path = Path("~/.my_project/config.json")

# expanduser() を呼ぶことで、自動的に現在のユーザーのホームディレクトリに展開される
# 例: ユーザー名が hanako の場合 -> /home/hanako/.my_project/config.json に変換される
resolved_path = config_path.expanduser()

# ログディレクトリの作成例
log_dir = Path("~/.my_project/logs").expanduser()
log_dir.mkdir(parents=True, exist_ok=True)

print(f"設定ファイルの場所: {resolved_path}")
```

`Path("~/.my_project/config.json").expanduser()` と記述することで、Pythonが自動的にOSの環境変数を読み取り、正しいフルパスに変換してくれます。この記法であれば、新PCに持っていっても一切コードを書き換える必要がありません。

## 2. Bashスクリプトの場合

LinuxやMac、WSLでよく使うシェルスクリプト（`.sh`）では、単純に `$HOME` という環境変数を参照するだけでOKです。

```bash
# 修正前（特定のユーザー名がパスに含まれている）
SCRIPT_PATH="/home/username/scripts/notify.sh"
LOG_DIR="/home/username/.claude/logs"

# 修正後（環境変数 $HOME を使用）
SCRIPT_PATH="$HOME/scripts/notify.sh"
LOG_DIR="$HOME/.claude/logs"

mkdir -p "$LOG_DIR"
```

シェルスクリプト内では `$HOME` と書くだけで、実行時にOSが自動的に現在のユーザーのホームディレクトリのパス（例: `/home/hanako` や `/Users/hanako`）に置き換えてくれます。

## 3. PowerShellの場合（Windows環境）

Windows環境でPowerShellを使っている場合は、`$env:USERPROFILE` という環境変数を利用します。

```powershell
# 修正後（Windowsの環境変数を使用）
$ConfigPath = "$env:USERPROFILE\.my_project\config.json"
```

このように言語ごとの「環境変数を展開する機能」を一度覚えてしまえば、あとは簡単です。

# 移行作業のフリクションを減らすメリット

このように「ハードコードされたパス」を「環境変数を用いた動的なパス」に書き換えるだけで、次のような多大なメリットが得られます。

1. **コピペで確実に動く**: 新しいPC環境に移行する際、ユーザー名が変わってもスクリプトや設定ファイルをそのまま複製するだけで即座に動作します。環境構築の自動化スクリプトなどを組んでいる場合、この対応が必須になります。
2. **Git管理の安全性**: 自分のローカルパス（実ユーザー名）がGitHubなどのリモートリポジトリにpushされることを防げるため、セキュリティ上のリスクも軽減できます。
3. **設定ファイルの共有化**: もしチームメンバーが同じツールやスクリプトを使いたいとなった場合でも、相手の環境に合わせてパスを書き換える手間を省くことができます。

# おわりに

開発環境の移行時につまずく「パスの問題」は、初心者の方が非常によく遭遇する壁です。
最初は自分のPCで動けば良いやと直書き（ハードコード）してしまいがちですが、`$HOME` 変数や `Path().expanduser()` を使う習慣をつけるだけで、将来の自分（と新しいPC）を大きく助けることになります。

「新PCで突然スクリプトが動かなくなった！」と焦ることがないよう、皆さんも手持ちのスクリプトや設定ファイルのパスを見直してみてはいかがでしょうか？
