## はじめに

ITエンジニアの転職活動において、面接で自分の実績をアピールする際、「開発期間」「パフォーマンス向上率」「チーム規模」といった具体的な数値を交えることは非常に重要です。数値を伴う説明は説得力を高め、技術的な深掘りの質問を引き出しやすくなります。

しかし、転職活動が長期化するにつれて、自己紹介スクリプト、経歴書、ポートフォリオのREADMEなど、複数の面接対策資料を並行して管理・更新することになります。その結果、「**資料ごとに数字が微妙に違う**」という問題が起こりがちです。

実際に私自身の面接対策用ドキュメント（`interview-prep`）でも、手動での転記ミスによる数字の不整合や、無意識の数字の誇張が発覚し、修正に追われた経験があります。

本記事では、この課題を解決するために「ポートフォリオ数値マスター」を**SSOT（Single Source of Truth：唯一信頼できる情報源）**として一元管理し、Pythonスクリプトを用いて各種面接資料へ自動反映する仕組みを解説します。手動作業を排除し、100%事実ベースで自信を持って面接に臨むための具体的なアプローチを紹介します。

---

## 1. 面接対策における「数字の不整合」の罠

なぜ面接資料の数字管理で失敗してしまうのでしょうか？それは、人間の記憶と手作業による転記に頼っているからです。

自分のGitHubのコミット履歴を見直した際、以下のような修正を行っていました。

- `fix: 全章の数字をSSOT正典(ポートフォリオ数値マスター)に一致`
- `fix: 自己紹介章の事実誤認を修正(NexusCore/OpenClaw混同・数字誇張)`

転職活動のストレスや焦りから、「開発期間は半年くらいだったか？」と曖昧な記憶で資料に書き込んでしまうと、後々「実際は約5ヶ月だった」というように事実とズレが生じます。

面接官が複数の提出資料を見比べたときに数字に一貫性がないと、「事実を歪めていないか？」と不信感を持たれるリスクがあります。エンジニアとして誠実さを示すためにも、実績データはシステム的に管理されているべきです。

## 2. SSOT（数値マスター）の設計と自動化スクリプト

解決策として、プロジェクトの事実や数値をまとめた「ポートフォリオ数値マスター」と呼ぶJSONファイルを1つ用意し、これをSSOTとします。そして、各章（30秒・1分・2分の自己紹介、技術用語の解説など）のMarkdownファイルは、このマスターから自動生成するワークフローを構築します。

### マスター（SSOT）の定義例

まず、事実となるデータをJSON形式で1つのファイルに集約します。

```json
// /path/to/portfolio_metrics.json
{
  "project_a": {
    "name": "OpenClaw",
    "dev_period": "約5ヶ月",
    "team_size": 4,
    "processing_time_reduction": "60%"
  }
}
```

### Pythonによる自動展開スクリプト

次に、Pythonを使って、マスターの値をMarkdownテンプレートに流し込むスクリプトを作成します。標準ライブラリの `string.Template` を使うことで、外部パッケージに依存せずにシンプルに実装できます。

```python
import json
from pathlib import Path
from string import Template

# 各種ファイルの汎用的なパス設定
master_data_path = Path("/path/to/portfolio_metrics.json")
template_path = Path("/path/to/interview_template.md")
output_path = Path("/path/to/output/interview_prep.md")

def generate_interview_doc():
    # 1. SSOT（ポートフォリオ数値マスター）から数値を読み込む
    with open(master_data_path, "r", encoding="utf-8") as f:
        metrics = json.load(f)
    
    # テンプレート展開用にフラットな辞書を作成
    # 例: {"project_a_dev_period": "約5ヶ月", ...}
    flat_metrics = {}
    for project, data in metrics.items():
        for key, value in data.items():
            flat_metrics[f"{project}_{key}"] = value

    # 2. テンプレートファイルを読み込む
    with open(template_path, "r", encoding="utf-8") as f:
        template_content = f.read()
    
    # 3. テンプレートのプレースホルダーを置換
    # 例: $project_a_dev_period -> 約5ヶ月
    template = Template(template_content)
    result = template.safe_substitute(flat_metrics)
    
    # 4. 面接資料として出力
    with open(output_path, "w", encoding="utf-8") as f:
        f.write(result)
    
    print("面接資料を正常に生成・更新しました。")

if __name__ == "__main__":
    generate_interview_doc()
```

テンプレートとなるMarkdownファイル（`interview_template.md`）には、あらかじめ変数を埋め込んでおきます。

```markdown
## 自己紹介（第10章）
私が直近で開発に携わった「$project_a_name」では、$project_a_dev_periodの期間、$project_size人チームで開発に従事しました。
結果として、処理時間を$project_a_processing_time_reduction短縮することに成功しています。
```

このスクリプトを実行するだけで、一瞬で事実と完全に一致した面接資料が生成されます。

## 3. 自動化による面接対策への3つのメリット

この仕組みを導入することで、IT転職の面接対策において以下のような大きなメリットが得られます。

### ① 事実ベースの自信と一貫性
「この数字、他の資料と違ったっけ？」という不安から解放されます。すべての資料が単一のJSONファイルをソースとしているため、数字に絶対的な一貫性が生まれます。面接本番でも、堂々と事実ベースで実績を語ることができます。

### ② メンテナンス性の向上
開発が進んで新しい実績（例：「処理速度を90%削減」など）が出たら、マスターJSONの1箇所だけを書き換えるだけで済みます。全12章に及ぶ面接資料を手動で検索して修正する手間はゼロになります。

### ③ 抽象論を防ぐ技術的振り返り
マスターにデータを書き出す作業自体が、自分自身の経験の棚卸しになります。「ノーコードツールの代わりに自前で実装した理由」や「ワークフローを自動化した背景」といった定性的な部分も整理されるため、抽象論に終始せず、深掘り質問に強い面接回答が準備できます。

## おわりに

ITエンジニアの転職活動は、自分自身のスキルや実績をプロダクトとして売り込むプロセスです。だからこそ、ソフトウェア開発のベストプラクティスである「SSOT」や「自動化」の考え方を、自分自身の面接対策に取り入れるのは非常に理にかなっています。

手動でのコピペによるミスや誇張表現を防ぎ、客観的な事実のみで構成された面接資料は、面接官との信頼関係を構築する強力な武器になります。

「面接資料の数字がごちゃごちゃになって管理できない」と悩んでいる方は、ぜひJSONと数行のPythonスクリプトでSSOTの仕組みを構築してみてください。保守性が劇的に上がり、本来注力すべき「面接での表現力」の練習に時間を割けるようになりますよ。
