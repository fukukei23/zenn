## はじめに

皆さんは、GitHub上のIssueを毎日チェックして状況を整理する「日次トリアージ」といった自動化スクリプトを作っていないでしょうか。

一見簡単そうに見える自動化ですが、「ターミナルを開いて手動で実行したときは動くのに、cronやCI経由で実行すると `command not found` になってしまう」というトラブルに直面した経験はないでしょうか。

今回は、GitHub CLI（`gh` コマンド）を使ってIssueを自動取得するスクリプトを作成していた際に発生した、**非ログインシェルでのコマンド未検出問題**の解決プロセスをご紹介します。

この記事を通じて、対話型シェルと非対話型シェルの環境変数の違いを理解し、どんな環境でも動く「Portable（持ち運び可能）」な堅牢なスクリプトを書くための基礎知識を身につけられます！

## なぜ非ログインシェルで `gh` が見つからないのか？

問題の根本原因は、シェルの「起動方法」の違いにあります。

普段、私たちがターミナルアプリを開いて操作する場合、OSは**「対話型ログインシェル」**を起動します。このとき、ユーザーの設定ファイル（`.bash_profile` や `.bashrc`、`.zshrc` など）が読み込まれ、`gh`コマンドをインストールしたディレクトリへのPATHが自動的に通されます。

一方で、cronによる定期実行やバックグラウンドスクリプト、CIパイプラインからスクリプトを実行する場合は**「非対話型非ログインシェル」**として扱われることが多くあります。
このモードでは、これらの設定ファイルが読み込まれないため、PATHが通っておらず、システムが `gh` コマンドの場所を見つけられずにエラーになってしまうのです。

## 解決策：スクリプト内でのPATH補完と環境変数の活用

この問題を解決する一番確実な方法は、**「スクリプト自身の中で、明示的に必要なPATHを通す」**ことです。

Pythonで外部コマンドを呼び出す際、その前段階で `os.environ` を操作してPATHを補完してあげることで、非ログインシェル環境でもコマンドが見つかるようになります。

以下は、環境変数 `PATH` に一般的なコマンドのインストール先を追加し、`gh` コマンドを呼び出すPythonスクリプトの改善例です。

```python
import os
import subprocess
from pathlib import Path

def fetch_issues():
    # 1. 現在の環境変数PATHを取得
    current_path = os.environ.get("PATH", "")
    
    # 2. 非ログインシェルで読み込まれない可能性のある汎用パスを明示的に追加
    additional_paths = [
        "/usr/local/bin",            # 汎用的なシステムパス
        "/opt/homebrew/bin",         # Apple Silicon環境のパス
        f"{Path.home()}/.local/bin"  # ユーザー環境変数を利用して動的に生成
    ]
    
    # 既存のPATHに追加（重複を避ける簡易ロジック）
    for p in additional_paths:
        if p not in current_path:
            current_path = f"{p}:{current_path}"
            
    os.environ["PATH"] = current_path

    # 3. 補完したPATHを元にghコマンドを安全に実行
    try:
        result = subprocess.run(
            ["gh", "issue", "list", "--json", "title,url"],
            capture_output=True,
            text=True,
            check=True
        )
        print("Issuesの取得に成功しました！")
        print(result.stdout)
    except FileNotFoundError:
        print("エラー: ghコマンドが見つかりません。インストールを確認してください。")
    except subprocess.CalledProcessError as e:
        print(f"ghコマンドの実行に失敗しました: {e.stderr}")

if __name__ == "__main__":
    fetch_issues()
```

このように記述しておくことで、対話型・非対話型に限らず、スクリプトが自ら実行環境を整えてからコマンドを叩くことができるようになります。

## さらに堅牢な「Portable」なスクリプトにするために

PATH補完の問題を解消した後、さらに「別のPC環境に移行してもそのまま動くスクリプト」にするためのリファクタリングを行いました。

よくある失敗として、スクリプト内に特定のPCのユーザー名をハードコードしてしまうケースがあります。特定のローカルパスを直書きしてしまうと、別のユーザー環境に持っていった瞬間に動かなくなってしまいます。

これを防ぐための基本ルールは**「パスのハードコードを避け、環境変数を活用する」**ことです。

Pythonであれば、先ほどのコード例にあった `Path.home()`（あるいは `Path().expanduser()`）を使うことで、実行環境のホームディレクトリを動的に取得できます。シェルスクリプトの場合も同様に、直書きを避け `$HOME` 変数を使う癖をつけると良いでしょう。

```bash
# ❌ 悪い例：特定の環境に依存するハードコード
SCRIPT_PATH="/path/to/specific/user/project/scripts/notify.sh"

# ✅ 良い例：環境変数を利用して汎用的に記述
SCRIPT_PATH="$HOME/project/scripts/notify.sh"
```

実際に私の管理している自動化設定用のリポジトリでも、設定ファイルやスクリプトに含まれるハードコードを除去し、すべて環境変数に置き換える修正を行いました。これにより、PCを買い替えたり、チームメンバーと設定を共有したりする際に、「スクリプトをコピーするだけでそのまま動く」という強力なポータビリティ（Portable性）を実現できています。

## おわりに

今回は、`gh` コマンドを用いたIssue取得スクリプトのトラブルを通じて、シェルの環境変数やPATH設定の重要性について解説しました。

- **非ログインシェルでは `.bashrc` などが読み込まれない場合がある**
- **スクリプト内で明示的にPATHを通すことで環境差異を吸収できる**
- **パスのハードコードを避け、環境変数を使うことでポータビリティが向上する**

自動化スクリプトを作る際、自分のPC環境（対話型シェル）で動くことだけを考えると、いざ自動実行したときに謎のエラーに直面しがちです。
「実行環境のシェルはどんな状態か？」を意識するだけで、エンジニアとしての環境構築スキルは一段上がります。皆さんもぜひ、自分の自動化スクリプトが堅牢かどうか、見直してみてはいかがでしょうか。
