# はじめに

テストカバレッジを向上させることは、ソフトウェアの品質を保つために非常に重要です。しかし、「とりあえずテストを書いてカバレッジを100%にすれば良い」というわけではありません。

オープンソースプロジェクトである**NexusCore**の開発において、私たちはテスト数を4,862から4,895へと増やし、特定のコアモジュールにおいてカバレッジ95%越えを達成しました。具体的には、`auth.py`（84% → 95%+）や `api_keys.py`（74% → 90%+）など、認証や権限に関わる重要なモジュールのカバレッジを大幅に向上させています。

本記事では、単にテストの数を増やしたのではなく、「何のテストを書くかより、どう壊すか」という考え方を中心に、フォールバック経路のテストや、長関数のリファクタリングなど、実務で行った具体的なアプローチをわかりやすく解説します。

# 1. 「何のテストを書くか」より「どう壊すか」

テストを書く際、多くの人はまず「正常系」をテストします。「正しいIDとパスワードを入れたらログインできる」といった具合です。もちろんこれは重要ですが、カバレッジを95%以上に引き上げ、本番環境での障害を未然に防ぐためには**「システムが想定外の状況でどう振る舞うか（どう壊れるか）」**をテストすることが不可欠です。

NexusCoreの開発では、以下のような「異常系・フォールバック（代替処理）経路」に焦点を当てました。

- 外部ライブラリ（JWTなど）がインポートエラーを起こした場合のフォールバック
- DBが未初期化の状態でアクセスされたときの安全な処理
- `secrets.json` が空文字だった場合の挙動

たとえば、`api/auth.py` のテストにおいて、JWT認証が正常に機能するかだけでなく、環境変数 `HAS_JWT=False` の設定下で正しく別の認証ルートへフォールバックするかを検証するテストを11個追加しました。このように「壊し方」「逃げ方」を漏れなくテストすることで、エッジケースでの致命的なバグを防ぐ堅牢な設計を実現しています。

# 2. SimpleNamespaceで複雑なMockを排除する

「異常系をテストする」となると、`unittest.mock` の `MagicMock` や `patch` を駆使して、無理やりエラーを発生させる複雑なコードを書いてしまいがちです。しかし、Mockが増えすぎると「テストコードは通るけど、一体何を検証しているのか分からない」という状態に陥りやすくなります。

NexusCoreでは、Pythonの標準ライブラリである `types.SimpleNamespace` を活用し、過度なMockの使用を排除するアプローチを取りました。

```python
from types import SimpleNamespace

def test_fallback_when_jwt_disabled():
    # JWTが利用できない環境のリクエストオブジェクトをSimpleNamespaceで簡易作成
    mock_request = SimpleNamespace(
        headers={},
        query_params={}
    )
    
    # HAS_JWT=False のフォールバック経路をテスト
    # （JWTライブラリが無いことを疑似的に再現）
    user = auth.require_auth(mock_request, has_jwt=False)
    
    # フォールバック時に期待されるデフォルトのゲストユーザーが返ること
    assert user.role == "guest"
```

`SimpleNamespace` を使えば、辞書型のように属性を動的に付与できるため、テストに必要な最低限の属性だけを持った軽量なダミーオブジェクトを瞬時に作れます。重いMockライブラリに依存せず、純粋にロジックの堅牢性を検証できるため、テストコードの可読性と保守性が格段に向上します。

# 3. テストを書く前に「テストしやすいコード」にリファクタリングする

カバレッジを上げるためのもう一つの重要なステップは、テスト対象のコード自体を「テストしやすい形」にリファクタリングすることです。

NexusCoreの最近のコミット履歴を見ると、カバレッジ向上のPRと並行して、大規模な「一気通貫リファクタリング」が行われていることがわかります。具体的には、以下のような長関数の分割です。

- 文字列組み立て5関数 (199行→57行、180行→51行など)
- API/ルーティング6関数 (159行→44行、133行→75行など)

1つの関数が長すぎると、中に多数のif文が絡み合い、あらゆる分岐を網羅するテストを書くのは非常に困難です。関数の責務を小さく分割し、さらに未使用のimport削除やハードコードされた定数の整理（クリーンアップ）を行ったことで、結果的にテストが書きやすくなり、カバレッジを限界まで引き上げることができました。

なお、UIコンポーネント（Gradio）のテストにおいては、`sys.modules` の隔離問題（Isolation Issue）によるテストの不安定さに悩まされました。このように外部ライブラリのシステム内部状態が影響しテスト環境の完全なモック化が難しいケースでは、無理にパスさせようとせず、`pytest` の `xfail`（失敗が期待されるテスト）としてマークし、テストスイート全体が落ちないように工夫を行いました。

# おわりに

NexusCoreでテスト数を4,862から4,895へ増やし、カバレッジ95%越えを達成したプロセスは、決して単なる数字合わせではありません。

「システムをどう壊すか」を徹底的に考え、`SimpleNamespace` を活用してテストコードをシンプルに保ち、テストしやすいように本筋のコードをリファクタリングする。この3つのサイクルがあって初めて、高いカバレッジと堅牢なシステムが手に入ります。

「とりあえずテストを書く」のではなく、「システムがどう壊れるかを考える」という視点を持つことで、皆さんのプロジェクトのテスト設計もさらに良いものになるはずです。テスト設計に迷ったら、ぜひ「異常系のフォールバック」から見直してみてください。
